## 2. 本アプリの機能とロジック（拡充が踏まえるべき現仕様）

> この節は新規執筆者・翻訳者・図作成者が **事実誤認なく拡充するための一次要約**。出典は `concepts/architecture.md`（973 行）/`concepts/bfts.md`/`concepts/memory.md`/`reference/configuration.md`/`reference/skills.md`。各サブシステムに〔ロジック要約 / 一次ソース doc / 拡充の方向〕を付す。

### 2.1 全体像 — エンドツーエンドの自律研究パイプライン

平文の研究ゴール（`experiment.md`、最短 3 行）から、調査 → 仮説 → 実験探索 → 評価 → 解析 → 作図 → 執筆 → 査読 → 再現性検証までを人手なしで回す。**production コードにドメイン知識を持たせず、すべての領域判断を実行時に LLM へ委譲する**のが設計の核心（`architecture.md` §Design Invariants）。

10 の論理ステップ:
`survey`（先行研究調査）→ `generate_ideas`（VirSci マルチエージェント討議で仮説・主指標を確定）→ **BFTS**（最良優先木探索で実験構成を探索）→ `LLMEvaluator`（査読者として scientific_score を付与）→ tree 解析 → 作図 → LaTeX 執筆 → LLM 査読 → 再現性検証。

- **一次ソース doc**: `concepts/architecture.md` §What ARI Does / §Full Data Flow、`index.html`（Describe→Survey→Experiment→Write→Analyze→Verify の 6 ステップ）
- **拡充の方向**: この 10 ステップを **1 枚のパイプライン図**（§4 WS-B / B3）にし、各ステップから対応 concept/guide/reference へリンクする「ハブ図」を作る。

### 2.2 BFTS（Branch-and-Frontier / Best-First Tree Search）

- **2 プール設計**: `pending`（実行待ち＝親から展開済み）と `frontier`（実行完了・未展開）。
- **単一子展開**: `expand()` は 1 回の呼び出しで子を 1 つだけ生成。兄弟スコア・祖先チェーン・木の多様性指標・既存子・現在 depth/`max_depth`・残ノード予算を文脈に与え重複を避ける。
- **永続フロンティア + 退役ルール**: 完了ノードは展開後もフロンティアに残り再展開可能。子が `_scientific_score` で親を上回る（Rule A）か、`max_expansions_per_node` 回展開済み（Rule B）になると親は **retire**。
- **多様性ボーナス**: 直近 20 実行で過少なラベルに `+0.05`（`my_count*2 ≤ max_count` 時）。
- **枝刈りは硬い閾値のみ**: `current_total ≥ max_total_nodes` / `depth ≥ max_depth` / `metrics._sterile is True`。LLM 判断はここに混ぜない。
- **リトライ無し**: 失敗ノードは再実行せず `debug` 子を生む。
- **ノードラベル**: `draft` / `improve` / `debug` / `ablation` / `validation` / `other`（未知ラベルは `other` に丸め `raw_label` で原文保持）。
- **一次ソース doc**: `concepts/bfts.md`、`architecture.md` §Module Reference（`orchestrator/bfts.py`）
- **拡充の方向**: pending/frontier 状態遷移と退役 A/B を **状態遷移図**化（B1）。`max_total_nodes` / `max_depth` / `max_expansions_per_node` / `ARI_PARALLEL` の関係を 1 表に整理。

### 2.3 ノード実行 — ReAct エージェントループ

各ノードは `AgentLoop.run(node, experiment)`（`ari/agent/loop.py`）で実行。

- LLM が MCP レジストリからツールを選び実行 → SLURM ジョブは **COMPLETED までステップ予算外で自動ポーリング** → stdout を読み実験コードを生成・投入 → 出力から指標を JSON 抽出。
- **プロンプト合成は root と child で分岐**: root は `make_metric_spec → survey → generate_ideas → executor` の `tool_sequence[0]` を必ず最初に呼ばせる。child はラベル別の 1 行タスク（improve/ablation/validation/debug/draft/other）＋祖先メモリ検索結果（prior knowledge）を与える。`node.eval_summary`（展開器が決めた具体方向）は **child プロンプトには直書きされず**メモリ検索経由で間接的に届く。
- **硬い制約 vs 柔らかい制約**: 他ノードのメモリ書込み禁止（硬）/兄弟メモリ閲覧禁止（硬）/`generate_ideas` は 1 回のみ（硬・`_suppress_tools`）/child は `survey` を呼ぶな（柔・prose のみ）/予算（柔ヒント＋硬い timeout・step cap）。
- **work_dir の CoW 継承と sterile 検出**: 子の work_dir は親をコピーして種付け。ただし `_OUTPUT_BLACKLIST`（`results.csv` / `slurm-*.out` / `metrics.json` / `*.log` / `node_report.json` 等）は継承せず、ソース・スクリプト・入力データのみ継承。実行後 `compute_files_changed` の sha256 差分が `added=modified=deleted=0` なら子を **sterile**（`_scientific_score=0.0`）化して枝刈り。
- **一次ソース doc**: `architecture.md` §Per-Node Prompt Composition / §work_dir inheritance、`guides/experiment_file.md`
- **拡充の方向**: 「ノード起動時にエージェントが見るもの」の 1 枚図と、硬/柔・truncation 上限表は既に充実 → **FAQ/トラブルシュート**（"子ノードが親と同じ数値を出す" → sterile/blacklist 機構へ誘導）に橋渡しリンクを追加。

### 2.4 評価器 LLMEvaluator と設定可能 4 層

ノード成果物（stdout・ログ・スクリプト）から `{has_real_data, metrics, scientific_score, comparison_found}` を抽出し、`_scientific_score` が BFTS ランキング・lineage 判断・ベスト選定を駆動。`default.yaml` で 4 層を独立に差し替え可能（既定値は従来挙動の no-op）。

| 層 | キー | 役割 | 値 |
|---|---|---|---|
| A 合成式 | `evaluator.composite` | 軸別スコア→スカラ化 | `harmonic_mean`(既定) / `arithmetic_mean` / `weighted_min` / `geometric_mean` |
| B フロンティア戦略 | `bfts.frontier_score` | LLM 選択不能時の決定的フォールバック順位 | `scientific_plus_diversity`(既定) / `scientific_only` / `depth_penalized`(λ) / `ucb_like`(c) |
| C 軸集合 | `evaluator.axis_mode` | 査読 LLM に渡す評価軸 | `dynamic`(既定: 汎用5軸+rubric+plan) / `legacy`(固定5軸) / `custom` |
| D 選択プロンプト | `bfts.select_prompt` / `expand_select_prompt` | select/expand の LLM プロンプト差し替え | `FilesystemPromptLoader` キー |

- **一次ソース doc**: `reference/configuration.md` §BFTS Evaluation Layers、`architecture.md`（`evaluator/llm_evaluator.py`）
- **拡充の方向**: §4 WS-C のクックブックに「HPC speedup を測る custom 軸」「bottleneck 重視の `weighted_min`」など **コピペ設定レシピ**を載せる。

### 2.5 メモリ（Letta バックエンド・祖先スコープ）

各ノードは **自分の祖先チェーンだけ**を読む（兄弟は不可）。v0.6.0 以降 Letta（旧 MemGPT）で実装。

- チェックポイント毎に専用エージェント。`ari_node_<hash>`（祖先スコープのアーカイバル）と `ari_react_<hash>`（フラットな ReAct トレース）の 2 コレクション。
- `search_memory` は `passages.search`（埋め込み検索, `embed_query=True`）で `top_k=max(overfetch, limit*40)` を取り、`ancestor_ids`/`ari_checkpoint`/`kind==node_scope` でローカル後段フィルタ。SQL substring の `passages.list` は自然文クエリで 0 件になるため**意図的に不使用**。
- **Copy-on-Write**: 書込み系ツールは `node_id ≠ $ARI_CURRENT_NODE_ID` を拒否 → 祖先エントリは兄弟間でバイト安定。
- **可搬性**: `memory_backup.jsonl.gz` を各チェックポイントに保存し `ari resume` 時に空 Letta へ自動復元 → `cp -r` + `resume` が成立。
- **一次ソース doc**: `concepts/memory.md`、`reference/skills.md`（`ari-skill-memory`）
- **拡充の方向**: 祖先スコープ図（B2）＋ CoW の「なぜ」を用語集（glossary）に短く定義。

### 2.6 plan/venue 契約・lineage 判断・サブ実験

- **plan（≒`idea.json[0].experiment_plan`）= 評価の具体**（測る指標/比較対象/ablation）。**venue（≒`reviewer_rubrics/<id>.yaml`）= 判定基準**（採点次元）。`ARI_RUBRIC` が venue を選び、Phase 3 の BFTS 採点軸と公開査読の基準を **同時に**切り替える。
- **lineage decision**: 合成スコアが停滞すると BFTS フックが LLM に `continue / switch_to_idea / fanout / terminate` を選ばせる。switch/fanout は子の `idea.json` に代替案を `_pinned:True` で先頭固定して再launch。
- **継承チャネル**: rubric（`ARI_RUBRIC` 伝播・継承）/ memory（祖先スコープ・継承）/ idea カタログ（`lineage.py` が `parent_run_id` を辿る・読取専用継承）/ plan（**既定で非継承**＝子は自分で書く＝ピボット自由）。
- **一次ソース doc**: `architecture.md` §Plan/Venue contract / §Inheritance、`reference/rubric_schema.md`
- **拡充の方向**: 「rubric を替えると採点と査読が連動する」点は誤解されやすい → glossary + cookbook に明示。

### 2.7 post-BFTS パイプライン（workflow.yaml 駆動・最大 16 ステージ）

BFTS 完了後、`nodes_tree.json` を入力に YAML 駆動でステージ実行（依存に応じ並列）。主な流れ:

1. `transform_data`（`ari-skill-transform`）: 木を BFS 走査し LLM がハードウェア/手法/知見/比較を抽出 → `science_data.json`（typed split: parameters/measurements/predictions/scores）。
2. `search_related_work`（`ari-skill-web`, stage1 と並列）→ `related_refs.json`。
3. `generate_figures`（`ari-skill-plot`）: 図ごとに `kind=plot`(matplotlib→PDF/PNG) か `kind=svg`(→ラスタ化) を LLM が自律選択。
4. `vlm_review_figures`（`ari-skill-vlm`）: 主図を VLM 採点、<0.7 なら作図へ最大 2 回ループバック。
5. `generate_ear`（EAR=Experiment Artifact Repository を node_report 駆動で決定的構築。code/data/figures/README/reproduce.sh/LICENSE。実験出力は同梱しない）。
6. `write_paper`（`ari-skill-paper`）: draft→LLM review→revise を最大 2 周、Semantic Scholar から BibTeX。
7. `review_paper`: rubric 駆動で N 人の独立 reviewer（`ARI_NUM_REVIEWS_ENSEMBLE`）、N>1 で **Area Chair メタ査読**集約。
8. `ear_curate` → `finalize_paper`（`\codeavailability{}`/`\codedigest{}`/`\coderef{}` 注入。digest が信頼アンカー）→ `ear_publish`（backend: ari-registry / local-tarball / gh / zenodo、既定 staged）。
- **一次ソース doc**: `architecture.md` §Full Data Flow（Stage 1–16）、`concepts/publication-lifecycle.md`、`reference/registry.md`
- **拡充の方向**: ステージ依存グラフを図化。EAR の「何を同梱し何を除外するか」を 1 表に（既存記述を guide へ昇格）。

### 2.8 ORS 再現性検証（PaperBench 互換・2 フェーズ）

v0.7.0 で旧 LLM 判定経路を **決定的 2 フェーズ**へ置換（arXiv:2504.01848 互換）。

- `ors_generate_rubric`（`ari-skill-replicate`）: 最終論文から TaskNode 木の PaperBench 形式 rubric を自動生成。`task_category` は閉語彙へ正規化、canonical JSON + paper digest を sha256 で凍結。
- **Phase 1** `run_reproduce`（`ari-skill-paper-re`）: `reproduce.sh` をサンドボックスで実行。選択順 `slurm`（sbatch + `ARI_SLURM_PARTITION`）→ `docker` → `apptainer` → `singularity` → `local`（`ARI_PHASE1_SANDBOX` で上書き）。`expected_artifacts` を照合。
- **Phase 2** `grade_with_simplejudge`: PaperBench SimpleJudge を rubric 葉に対し N 回（既定 3）走らせ重み付き集約。**負例コントロール**（空リポ + 自明 reproduce.sh）が <5% でないと不合格 → 「無作業を褒めない」ことを担保。
- **paper-audit モード**（v0.7.2）: 同じ rubric 機構を反転し「論文自体が再現に足る記述か」を会場条件テンプレ（sc / neurips / nature）で監査。
- **3 ステージ bridge 契約**（v0.8.0）: `rollout_submission` / `reproduce_submission` / `judge_submission` を vendor プロトコル準拠で実装。env-truth ガードレール（probe-before-scaffold 等）。
- **一次ソース doc**: `guides/paperbench/*`、`reference/api_paperbench.md`、`reference/rubric_schema.md`、`reference/execution_profile.md`
- **拡充の方向**: Phase1 サンドボックス選択フローの決定木図。**bridge/agent-facing テキストに dogfood 数値・固有論文名・クラスタ module 名を焼き込まない**方針を執筆ガイドに明記（既存方針）。

### 2.9 アーキテクチャ層・MCP スキル・状態モデル

- **5 層構成**（0 primitives → 1 domain models → 2 orchestrator → 3 agent → 4 pipeline → 5 entry points）。スキルは `ari.public.*` のみ import 可（`test_public_api_boundary.py` が CI 強制）。
- **MCP スキル 14 個**（13 default + 1 additional `ari-skill-orchestrator`）。core は async MCP client でツールを dispatch。
- **状態は完全にチェックポイントスコープ**（v0.5.0 で `$HOME/.ari/` を撤廃）。`checkpoints/{YYYYMMDDHHMMSS_slug}/` 配下に `tree.json`/`nodes_tree.json`/`science_data.json`/`full_paper.tex`/`ear/`/`repro/`/`settings.json`/`memory_backup.jsonl.gz` 等。`PathManager` が唯一の真実源。**API キーは `settings.json` に保存せず** `.env`（探索順 checkpoint→ARI root→ari-core→home）か環境変数から読む。
- **一次ソース doc**: `architecture.md` §File Structure / §Module Reference / §Layered architecture、`reference/skills.md`、`reference/environment_variables.md`、`guides/extension_guide.md`
- **拡充の方向**: チェックポイント・レイアウト図、5 層依存図、用語集（MCP/EAR/ORS/CoW/frontier 等）。

> **運用上の注意（拡充作業時）**: 状態がチェックポイントスコープである以上、ドキュメントの例も **出力先を `workspace/checkpoints/<ts>_<slug>/` に揃える**（`$HOME` 直下や `/tmp` は不可）。GUI ポートは 8765、起動/停止は `start.sh`/`shutdown.sh` に準拠。

## 3. ギャップ分析（`docs/` を読んで判明した不足）

### G1. Tutorial 層が薄い（最優先）
Diátaxis の "Tutorial" カテゴリが `getting-started/quickstart.md` の **1 本のみ**。FAQ・用語集（glossary）・getting-started のランディング/索引・"2 本目のチュートリアル"（最後まで通す題材ベースの手引き）が無い。初学者が quickstart の次に進む明示的な経路が欠落。

### G2. 概念ドキュメントに図が無い
`concepts/architecture.md`（973 行）/`bfts.md`/`memory.md` はテキストと ASCII のみ。§2 で挙げた BFTS 2 プール遷移・祖先チェーン memory・3 ステージ bridge は **図示で理解が大きく改善**する候補。Mermaid 等の導入余地。

### G3. `about/` が薄い
`about/release_policy.md` のみ。リポジトリ直下に `CONTRIBUTING.md` は存在するが docs からの導線が弱い。`SECURITY.md`・互換性/サポートポリシー・ライセンス・CHANGELOG への索引が docs 側に無い。

### G4. 翻訳の"内容"ドリフト検知が無い（構造的リスク）
パリティ表と `check_doc_*` は **ファイルの存在とリンク** しか見ない。en を更新しても ja/zh が古いまま、を機械的に検知する仕組みが無い。`last_verified` は現状 3 言語とも 2026-05-25 で揃うが、**今後の更新で必ずドリフトする**。「en の `last_verified` ≦ 翻訳の `last_verified`」を検査するゲートが未整備。

### G5. ナビゲーション / サイト UX
- `.md` 間に next/prev・パンくずが無く、`README.md` が唯一の目次（SoT）。
- 静的 HTML サイト（`docs.html`）に**全文検索が無い**。
- i18n は `data-lang` の表示切替に依存し、ディープリンク時の言語保持が弱い。

### G6. 実例 / クックブック不足
`reference/configuration.md`（714 行）・`guides/experiment_file.md` は網羅的だが、§2.4 の設定可能 4 層や §2.8 のサンドボックス選択など **コピペで動く完結レシピ集**（laptop / HPC / cloud プロファイル別、PaperBench 各モード別）が無い。

### G7. HTML サイトと `.md` の二重持ち

## 7. リスクと留意点

- **ゲート遵守が最大の制約**: `docs/` 配下の新 `.md` は front-matter 無しだと `check_doc_sources.py` の coverage に必ず引っかかる。新規 doc は必ず front-matter・パリティ表・3 言語をセットで用意する。
- **二重管理（G7）**: `docs.html` を触る変更は canonical `.md` との整合を必ず確認。理想は HTML 側を要約+リンクに寄せる。
- **エージェント向けテキストの中立性**: 図や例に dogfood 数値・固有論文名・クラスタ固有の module 名を埋め込まない（bridge は paper/cluster-agnostic）。§2.8 参照。
- **ドメイン知識ゼロ原則を崩さない**: ドキュメントで「ARI が X 分野用に最適化されている」等と書かない。領域判断は実行時 LLM が行う（§2.1 / §2.9 Design Invariants）。
- **新ゲート（D1）の段階導入**: いきなり error 化すると既存の僅かな差で CI が落ちうる。warning から開始し、全 doc を揃えてから error 昇格。
- **G8 は docs スコープ外**: `ari-skill-review/` の掃除はドキュメント PR に混ぜず独立 PR で。

- 削除は **専用コミット**で行い、メッセージ末尾に `[plan-deletion]` を付して既存慣習に揃える。
