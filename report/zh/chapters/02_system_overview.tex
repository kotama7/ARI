\section{系统概览}\label{sec:overview}

\subsection{完整堆栈}\label{sec:overview:layout}

\begin{figure*}[t]
  \centering
  \resizebox{0.95\textwidth}{!}{\input{../shared/figures/tikz/F10_full_stack.tex}}
  \caption{v0.8.0 时 ARI 的完整堆栈。CLI / SDK 入口驱动单一编排
           器(\texttt{ari-core});编排器拥有三个引擎(BFTS、谱
           系遍历器、ReAct \texttt{AgentLoop}),并经 MCP 层向
           14 技能注册表分派。所有持久化都落在单一检查点目录:
           成本账本、\texttt{tree.json}、\texttt{results.json}。
           实线为控制,虚线为状态。}
  \label{fig:fullstack}
\end{figure*}

\Cref{fig:fullstack} 描绘了 v0.8.0 的 ARI 完整堆栈。系统分为
一个驱动器与多个工具提供方。驱动器(\texttt{ari-core})拥有
编排器、BFTS 引擎、谱系遍历器、智能体循环、检查点、成本追踪
器与配置加载器。每个技能为独立 Python 包,以\term{模型上下文协议}
(Model Context Protocol, MCP)服务器形式发布,并附自身
\texttt{skill.yaml} 声明。MCP 是 Anthropic 标准化的
JSON-RPC-over-stdio 接口,用于将工具式能力暴露给 LLM 代理循环;
ARI 以此作为 \texttt{ari-core} 与各技能之间的统一接缝。当前在树的 14 个技
能:\texttt{benchmark}、\texttt{coding}、
\texttt{evaluator}、\texttt{hpc}、\texttt{idea}、
\texttt{memory}、\texttt{orchestrator}、\texttt{paper}、
\texttt{paper-re}、\texttt{plot}、\texttt{replicate}、
\texttt{transform}、\texttt{vlm}、\texttt{web}。

编排器只负责调用分派,不执行领域工作。这种分离至关重要:技能
可被增删或替换而不必触动搜索算法 —— BFTS 引擎并不知道哪些技能
参与运行;一切由 MCP 分派层中介。

\Cref{fig:architecture} 是本报告其余部分使用的「运营视图」
(控制 + 状态输出);\cref{fig:fullstack} 是「分层视图」(驱
动器 → MCP → 技能 → 持久化)。

\subsection{流水线 DAG}\label{sec:overview:pipeline}

\begin{figure}[t]
  \centering
  \resizebox{\columnwidth}{!}{\input{../shared/figures/tikz/F05_pipeline_dag.tex}}
  \caption{流水线 DAG:idea $\to$ exploration $\to$ paper $\to$
           review,由 \term{改稿} 闭合反向边。虚线输出持久化至
           检查点;改稿循环重写草稿并重跑评审。}
  \label{fig:pipeline}
\end{figure}

一次运行可归约为四个工位的流水线 (\cref{fig:pipeline})。idea
工位选取或生成研究问题;exploration 工位在 BFTS 下扩展节点;
paper 工位将最佳谱系汇编为草稿;review 工位以评分细则评估草
稿。该图为 DAG 加上一条由 review 回到 paper 的反向边:仅此边
可能引入非单调性(改稿可能恶化某一轴),而\term{收敛标准}
(\cref{sec:paper:loop})则是限定反向边数量的约束。

流水线由编排器中的单一运行器驱动;科学数据组装器弥合探索输出与
论文输入之间的间隙,汇集撰写器所需的按节点产物。

\subsection{检查点作用域与设计原则}\label{sec:overview:principles}

ARI 围绕五条\term{设计原则} (P1--P5) 构建。最具约束力者为
\textbf{P2:确定性}。每一步随机化均记录其种子;每次 LLM 调用均
记录模型身份、提示与输出;每次工具调用都以其节点身份标记
(写时复制护栏)以确保写入落在所属节点的工作目录内。重
跑检查点必须再现相同的产物,字节级一致,除非字段被显式标记为
时钟相关或模型端非确定。

其余原则简述如下:

\begin{itemize}
  \item \textbf{P1 单一产物树}:运行的全部输出位于
        \texttt{\$ARI\_CHECKPOINT\_DIR} 之下,不外溢至
        \texttt{\$HOME} 或 \texttt{/tmp}。
  \item \textbf{P3 小组件}:每个技能为独立包;替换技能不需重建
        \texttt{ari-core}。
  \item \textbf{P4 显式谱系}:每个节点与每个检查点均记录其父级,
        派生实验仅须重算其差量。
  \item \textbf{P5 成本透明}:成本追踪器为每次模型调用附加美元
        成本,使运行预算可控。
\end{itemize}

检查点序列化器写出三个顶层文件:完整搜索树写入
\texttt{tree.json},按节点的载荷写入
\texttt{nodes\_tree.json},运行末聚合写入
\texttt{results.json}。跨检查点的谱系遍历由子向祖先攀登。
