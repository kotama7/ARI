\section{系统概览}\label{sec:overview}

\subsection{完整堆栈}\label{sec:overview:layout}

\begin{figure*}[t]
  \centering
  \resizebox{0.95\textwidth}{!}{\input{../shared/figures/tikz/F10_full_stack.tex}}
  \caption{v0.9.0 时 ARI 的完整堆栈,分为五层:入口;
           \texttt{ari-core} 编排器(徽标~1);引擎层 ——
           BFTS 引擎与 ReAct \texttt{AgentLoop} 在此构成迭代
           循环(徽标~2),谱系遍历器位于其旁;MCP 分派枢纽
           (徽标~3),向技能注册表的四个生命周期列扇出;以及
           检查点持久化。实线为控制,虚线为次级流(谱系查询、
           持久化)。}
  \label{fig:fullstack}
\end{figure*}

\Cref{fig:fullstack} 将 v0.9.0 的 ARI 完整堆栈画为五层;本节自
顶向底逐层讲解。系统分为一个驱动器与多个工具提供方:驱动器
(\texttt{ari-core})拥有编排器、搜索与智能体引擎、谱系遍历器、
检查点序列化器、成本追踪器与配置加载器,而每一项领域能力 ——
编写代码、运行基准、评审草稿 —— 都驻留在一个技能之中。

\paragraph{入口。}
一次运行从命令行界面(CLI)、公共 Python 界面(软件开发工具包,
SDK)或可视化服务器启动 —— 后者是一个 HTTP 仪表盘,向浏览器
暴露实时搜索状态、按检查点的文件以及启动控制。所有路径都汇聚到
同一个编排器入口(徽标~1);交互式使用并不会走第二条代码路径。

\paragraph{引擎层。}
\term{最佳优先树搜索}(best-first tree search, BFTS)引擎维护
由研究步骤节点构成的树,并决定下一步运行哪个节点以及如何扩展它
(\cref{sec:exploration}) —— 这是一种「在候选方案上搜索」的框架,
ARI 与 AIDE 共享此构想 \cite{jiang2025aide}。每次扩展都把一个节点
交给 \texttt{AgentLoop} —— 一个 Reason-and-Act(ReAct)智能体
\cite{yao2023react},它将自由文本推理与工具调用交织进行,直到该
节点的工作完成:引擎提供方向,智能体提供执行 —— 即一条迭代循环
(徽标~2)。位于循环旁的谱系遍历器是被咨询而非被驱动的:它跨检查点
攀爬父指针,使派生运行得以继承其祖先的想法池与上下文
(\cref{sec:exploration:lineage})。

\paragraph{MCP 分派。}
每次工具调用都要穿过分派层(徽标~3)。\term{模型上下文协议}
(Model Context Protocol, MCP)是 Anthropic 标准化的
JSON-RPC-over-stdio 接口,用于将工具式能力暴露给语言模型智能体
循环;ARI 以此作为 \texttt{ari-core} 与每个技能之间的统一接缝。
每个技能都是一个独立的 Python 包,其服务器以池化子进程运行,且
当技能自带解释器时便在其自身解释器下运行。技能通过一条配置项接入,
该项指明其包路径以及其工具被暴露的流水线阶段;工具管理器按当前
活动阶段过滤工具,因此探索节点无法调用论文撰写工具
(\cref{sec:exploration:react})。

当前在树的十四个技能包为:\texttt{benchmark}、\texttt{coding}、
\texttt{evaluator}、\texttt{hpc}、\texttt{idea}、\texttt{memory}、
\texttt{orchestrator}、\texttt{paper}、\texttt{paper-re}、
\texttt{plot}、\texttt{replicate}、\texttt{transform}、
\texttt{vlm} 与 \texttt{web}。\cref{fig:fullstack} 中的注册表按
生命周期阶段将其工具面分组:生成(idea、coding)、评估(evaluator、
review)、撰写与复现(paper、replicate、paper-re),以及基础设施
与输入输出 —— 其中 \texttt{hpc} 技能是通往集群批处理调度器的接缝,
使实验可作为派发作业运行。有两个方框值得一提:\emph{review} 命名的
是一个工具面而非一个包(草稿评审工具由 evaluator 与 paper 技能
托管;\cref{sec:paper:loop}),而 \texttt{orchestrator} 技能经
MCP 暴露运行管理,使 ARI 自身可作为工具被外部智能体驱动。

编排器分派调用,但自身不执行任何领域工作。这种分离至关重要,因为
技能可被增删或替换而不必触动搜索算法:BFTS 引擎并不知道哪些技能
参与运行。

  \centering
  \resizebox{\columnwidth}{!}{\input{../shared/figures/tikz/F01_architecture.tex}}
  \caption{运营视图。编排器经两条分派总线向三个生命周期列
           (填充编码阶段)以及无阶段的支撑技能扇出。虚线为状态:

最底层是持久化:一次运行产生的每一个字节都落入单一检查点目录,
与序列化的树及成本账本并列(\cref{sec:overview:principles})。
\cref{fig:architecture} 中较小的图是本报告其余部分使用的「运营」
视图(仅控制总线与状态边);\cref{fig:fullstack} 则是「分层」
视图(驱动器 $\to$ 引擎 $\to$ MCP $\to$ 技能 $\to$ 持久化)。

\subsection{流水线 DAG}\label{sec:overview:pipeline}

\begin{figure}[t]
  \centering
  \resizebox{\columnwidth}{!}{\input{../shared/figures/tikz/F05_pipeline_dag.tex}}
  \caption{流水线 DAG:两个带标题 \emph{search} 与 \emph{write
           \& review} 之下的四个工位(徽标 1--4)。每个工位将其
           产物持久化至检查点(虚线)。唯一着重标示的边是
           \term{改稿},即重写草稿并重跑评审的反向边。}
  \label{fig:pipeline}
\end{figure}

一次运行可归约为一条四工位的流水线(\cref{fig:pipeline})。idea
工位选取或生成研究问题;exploration 工位在 BFTS 下扩展节点;
paper 工位将最佳谱系汇编为草稿;review 工位以评分细则评估草稿。
该图是一个 DAG 加上一条由 review 回到 paper 的反向边:仅此边可能
引入非单调性(改稿可能恶化某一轴),而\term{收敛标准}
(\cref{sec:paper:loop})则限定反向边的数量。流水线由编排器中的
单一运行器驱动;一个科学数据组装器汇集撰写器所需的按节点产物
(\cref{sec:paper:evidence})。

撰写与评审带隐藏着一条固定的\emph{门链},草稿在定稿前必须通过它。
撰写之后,草稿的论断经稳定标识符与已记录的证据相链接;一个确定性的
论断—证据门(无语言模型)在容差范围内由已记录的结果重新推导出每一个
所报告的数字;一个互补的语义评审标记过度声称并喂入改稿循环;改稿之后,
链接、评审与门会再运行一次,作为最终核查 —— 在严格策略下,只要存在
任何客观不符,该核查便阻断定稿(\cref{sec:paper:contract})。阻断门
是经一层薄技能包装器抵达的 core 侧代码,因此替换某个技能并不能削弱它;
阻断仅保留给客观谬误,而主观评审结论引导改稿器,却绝不对发布设门。
该门链将 Story2Proposal 的契约穿线式生成以执行落地的形式实现
\cite{qian2026story2proposal}。

分支隔离得以保持(\cref{sec:exploration:lineage-chaining})。

\subsection{检查点作用域与设计原则}\label{sec:overview:principles}

ARI 围绕五条\term{设计原则}(P1--P5)构建。最具约束力者为
\textbf{P2:确定性}。每一步随机化均记录其种子;每次 LLM 调用均记录
其模型身份、提示与输出;每次工具调用都以其节点身份标记(一道写时复制
护栏),以确保其写入落在所属节点的工作树之内。重跑一个检查点必须
逐字节再现相同的产物,除非字段被显式标记为时钟相关或模型端非确定。

其余原则简述如下:

\begin{itemize}
  \item \textbf{P1 单一产物树}:一次运行的每一个输出都位于
        \texttt{\$ARI\_CHECKPOINT\_DIR} 之下。不外溢至
        \texttt{\$HOME} 或 \texttt{/tmp}。
  \item \textbf{P3 小组件}:每个技能都是独立的包;替换某个技能
        不需重建 \texttt{ari-core}。
  \item \textbf{P4 显式谱系}:每个节点与每个检查点都记录其父级,
        因此派生实验只须重算其差量。
  \item \textbf{P5 成本透明}:成本追踪器为每次模型调用附加美元
        成本,使用户得以为一次运行设界;技能子进程追加到同一条
        运行级账本。
\end{itemize}

检查点序列化器写出三个顶层文件:完整搜索树写入 \texttt{tree.json},
供论文流水线消费的轻量按节点导出写入 \texttt{nodes\_tree.json},
运行末聚合写入 \texttt{results.json}。跨检查点的谱系遍历经由每个
检查点记录在其运行元数据中的父指针,自子节点向祖先攀爬。

  \centering
  \resizebox{\columnwidth}{!}{\input{../shared/figures/tikz/F13_checkpoint_layout.tex}}
  \caption{检查点目录布局。运行级文件(左带)由编排器写入;按节点
           工作树(右带)保存一次 BFTS 扩展内产生的产物,包括每个

  \label{fig:checkpoint}

\Cref{fig:checkpoint} 描绘检查点的磁盘形状:左侧为共享运行级文件,
右侧为按节点工作树。一个节点的工作树起初是其父级工作树的副本,因此
每棵树都是该步骤上实验的自包含快照,而为证据组装奠基的自报告
(\cref{sec:paper:evidence})就紧挨着它所描述的产物。该形状即契约:
任何把 ARI 输出当作输入的工具 —— 后续运行、回归检查、论文草稿 ——
都指向单一这样的目录。
