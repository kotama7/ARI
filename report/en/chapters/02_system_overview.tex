\section{System Overview}\label{sec:overview}

\subsection{The full stack}\label{sec:overview:layout}

\begin{figure*}[t]
  \centering
  \resizebox{0.95\textwidth}{!}{\input{../shared/figures/tikz/F10_full_stack.tex}}
  \caption{The full ARI stack at v0.9.0 as five layers: entry points;
           the \texttt{ari-core} orchestrator (badge~1); the engine
           layer, where BFTS engine and ReAct \texttt{AgentLoop} form
           the iteration loop (badge~2) with the lineage walker
           beside it; the MCP dispatch hub (badge~3) fanning into the
           skill registry's four lifecycle columns; and checkpoint
           persistence. Solid edges are control; dashed edges are
           secondary flows (lineage lookup, persistence).}
  \label{fig:fullstack}
\end{figure*}

\Cref{fig:fullstack} draws the full ARI stack at v0.9.0 as five
layers; this section walks them top to bottom. The system splits
into one driver and many tool providers: the driver
(\texttt{ari-core}) owns the orchestrator, the search and agent
engines, the lineage walker, the checkpoint serialiser, the cost
tracker, and the configuration loader, while every domain capability
--- writing code, running benchmarks, reviewing drafts --- lives in
a skill.

\paragraph{Entry points.}
A run starts from the command-line interface (CLI), from the public
Python interface (the software development kit, SDK), or from the
visualisation server --- an HTTP dashboard exposing live search
state, per-checkpoint files, and launch controls to a browser. All
paths converge on the same orchestrator entry (badge~1); interactive
use does not get a second code path.

\paragraph{Engine layer.}
The \term{best-first tree search} (BFTS) engine maintains the tree
of research-step nodes and decides which node to run next and how
to expand it (\cref{sec:exploration}) --- a
search-over-candidate-solutions framing ARI shares with AIDE
\cite{jiang2025aide}. Each expansion hands a node to the
\texttt{AgentLoop}, a Reason-and-Act (ReAct) agent
\cite{yao2023react} that interleaves free-text reasoning with tool
calls until the node's work is done: the engine supplies direction,
the agent execution --- one iteration loop (badge~2). The lineage
walker beside the loop is consulted rather than driven: it climbs
parent pointers across checkpoints so a derivative run can inherit
its ancestors' idea pool and context
(\cref{sec:exploration:lineage}).

\paragraph{MCP dispatch.}
Every tool call crosses the dispatch layer (badge~3). The
\term{Model Context Protocol} (MCP) is the JSON-RPC-over-stdio
interface that Anthropic standardised for exposing tool-style
capabilities to a language-model agent loop; ARI uses it as the
universal seam between \texttt{ari-core} and every skill. Each skill
is a separate Python package whose server runs as a pooled child
process, under the skill's own interpreter when it ships one. A
skill is wired in by a configuration entry naming its package path
and the pipeline phases in which its tools are exposed; the tool
manager filters tools by the active phase, so an exploration node
cannot call the paper-writing tools (\cref{sec:exploration:react}).

The fourteen skill packages currently in tree are:
\texttt{benchmark}, \texttt{coding}, \texttt{evaluator},
\texttt{hpc}, \texttt{idea}, \texttt{memory}, \texttt{orchestrator},
\texttt{paper}, \texttt{paper-re}, \texttt{plot}, \texttt{replicate},
\texttt{transform}, \texttt{vlm}, and \texttt{web}. The registry in
\cref{fig:fullstack} groups their tool surfaces by lifecycle phase:
generation (idea, coding), evaluation (evaluator, review), writing
\& replication (paper, replicate, paper-re), and infrastructure and
I-O --- where the \texttt{hpc} skill is the seam to a cluster batch
scheduler, so experiments can run as dispatched jobs. Two boxes
deserve a note: \emph{review} names a tool surface rather than a
package (draft-review tools are hosted by the evaluator and paper
skills; \cref{sec:paper:loop}), and the \texttt{orchestrator} skill
exposes run management over MCP, so ARI itself can be driven as a
tool by an external agent.

The orchestrator dispatches calls but performs no domain work
itself. The separation matters because skills can be added or
swapped without touching the search algorithm: the BFTS engine is
unaware of which skills participate.

The bottom layer is persistence: every byte a run produces lands in
a single checkpoint directory, alongside the serialised tree and the
cost ledger (\cref{sec:overview:principles}). The smaller diagram in
\cref{fig:architecture} is the ``operational'' view used in the rest
of the report (control buses and state edges only);
\cref{fig:fullstack} is the ``layered'' view (driver $\to$ engines
$\to$ MCP $\to$ skills $\to$ persistence).

\subsection{Pipeline DAG}\label{sec:overview:pipeline}

\begin{figure}[t]
  \centering
  \resizebox{\columnwidth}{!}{\input{../shared/figures/tikz/F05_pipeline_dag.tex}}
  \caption{Pipeline DAG: four stations (badges 1--4) under two band
           headings, \emph{search} and \emph{write \& review}. Each
           station persists its artefact to the checkpoint (dashed).
           The single accented edge is \term{refine}, the back edge
           that rewrites the draft and re-runs the review.}
  \label{fig:pipeline}
\end{figure}

A run reduces to a four-station pipeline (\cref{fig:pipeline}). The
idea station selects or generates research questions; the
exploration station spawns nodes under BFTS; the paper station
compiles the best lineage into a draft; and the review station
grades the draft against the rubric. The graph is a DAG plus one
back edge from review to paper: only this edge can introduce
non-monotonicity (refining can worsen an axis), and the
\term{convergence criterion} (\cref{sec:paper:loop}) bounds the
back-edge count. The pipeline is driven by a single runner in the
orchestrator; a scientific-data assembler collates the per-node
artefacts the writer needs (\cref{sec:paper:evidence}).

The write-and-review band hides a fixed \emph{gate chain} the draft
must clear before finalisation. After writing, the draft's claims
are linked to recorded evidence by stable identifiers; a
deterministic claim--evidence gate (no language model) re-derives
every reported number from the recorded results within a tolerance;
a complementary semantic review flags overclaims and feeds the
refine loop; and after refinement the linking, the review, and the
gate run once more as a final check that, under the strict policy,
blocks finalisation while any objective mismatch stands
(\cref{sec:paper:contract}). The blocking gate is core-side code
reached through a thin skill wrapper, so swapping a skill cannot
weaken it; blocking is reserved for objective falsehoods, while
subjective review findings steer the refiner but never gate
publication. The chain realises Story2Proposal's contract-threaded
generation in execution-grounded form \cite{qian2026story2proposal}.

  \caption{BFTS control flow: one step as a spine of five numbered

           own prompt file (dashed, above); when the parse gate
           rejects the returned index, selection falls back to the
           deterministic \texttt{\_fallback\_score} (scientific score
           plus \texttt{diversity\_bonus}). Below the spine: the
           v0.9.0 lineage-chaining hints into the expansion, and the
           stagnation detector into the prune gate. The accented edge
           appends the child to the frontier; the dotted margin path
           continues to the next iteration.}
  \label{fig:bftsflow}
\end{figure*}

\Cref{fig:bftsflow} traces a single BFTS step end-to-end. The
implementation is laid out in \cref{sec:exploration}; the diagram
fixes the vocabulary used by the rest of the report (frontier,
selection, fallback, expand, prune, stagnation) and shows the prompt
files behind the two LLM-driven stations (\texttt{bfts\_select.md},
\texttt{bfts\_expand.md}); a third prompt,
\texttt{bfts\_expand\_select.md}, drives the choice of \emph{which}
completed node to expand in the full loop
(\cref{sec:exploration:node-selection}). The two hint boxes below
the spine are new at v0.9.0 and implement lineage chaining for
computed evidence: the coverage hint steers the expansion target
toward the node already holding the required measurements, and the
lineage hint tells the new child which inherited measurement files
it holds. The hints carry only names, never values, so branch
isolation is preserved (\cref{sec:exploration:lineage-chaining}).

\Cref{fig:paperre} is the second-level view of \texttt{paper-re},
the skill that carries the PaperBench-protocol reproduction path
\cite{starace2025paperbench}. ARI's contribution at this layer is
small but specific: a thin adapter wraps each agent tool so its
output is size-bounded, and redirects the vendored solver's
hard-coded paths to ARI's per-node workspace; everything else flows
through the PaperBench upstream verbatim, which keeps ARI aligned
with the upstream replication-task framing and grade semantics.
\Cref{sec:paper:paperbench} develops the three-stage
rollout--reproduce--grade contract the bridge drives.

\subsection{Checkpoint scoping and the design principles}\label{sec:overview:principles}

ARI is built around five \term{design principles} (P1--P5). The
binding one is \textbf{P2: determinism}. Every randomised step
records its seed; every LLM call records its model identity, prompt,
and output; every tool call is tagged with its node identity (a
copy-on-write guard) so its writes land inside the right node's
working tree. Re-running a checkpoint must reproduce the same
artefacts byte-for-byte, except for fields explicitly marked as
wallclock or as model-side non-determinism.

The other principles, in summary:

\begin{itemize}
  \item \textbf{P1 single artefact tree}: every output of a run lives
        below \texttt{\$ARI\_CHECKPOINT\_DIR}. Nothing leaks to
        \texttt{\$HOME} or \texttt{/tmp}.
  \item \textbf{P3 small components}: each skill is a separate
        package; replacing one skill does not require rebuilding
        \texttt{ari-core}.
  \item \textbf{P4 explicit lineage}: every node and every checkpoint
        record their parent, so a derivative experiment can recompute
        only its delta.
  \item \textbf{P5 cost transparency}: the cost tracker attaches
        dollar costs to every model call so the user can bound a run;
        skill subprocesses append to the same run-level ledger.
\end{itemize}

The checkpoint serialiser writes three top-level files: the full
search tree to \texttt{tree.json}, a lightweight per-node export
consumed by the paper pipeline to \texttt{nodes\_tree.json}, and
end-of-run aggregates to \texttt{results.json}. Lineage walks across
checkpoints climb from a child to its ancestors through the parent
pointer each checkpoint records in its run metadata.

  \caption{Checkpoint directory layout. Run-level files (left band)

\Cref{fig:checkpoint} draws the on-disk shape of a checkpoint:
shared run-level files on the left, per-node working trees on the
right. A node's working tree starts as a copy of its parent's, so
each tree is a self-contained snapshot of the experiment at that
step, and the self-report that grounds evidence assembly
(\cref{sec:paper:evidence}) sits next to the artefacts it describes.
The shape is the contract: any tool that takes ARI's output as input
--- a follow-up run, a regression check, a paper draft --- points at
a single such directory.
