\section{System Overview}\label{sec:overview}

\subsection{The full stack}\label{sec:overview:layout}

\begin{figure*}[t]
  \centering
  \resizebox{0.95\textwidth}{!}{\input{../shared/figures/tikz/F10_full_stack.tex}}
  \caption{The full ARI stack at v0.8.0. The CLI / SDK entry-points
           drive a single orchestrator (\texttt{ari-core}); the
           orchestrator owns three engines (BFTS, lineage walker,
           ReAct \texttt{AgentLoop}) and dispatches into the
           14-skill registry through the MCP layer. All persistence
           lands inside one checkpoint directory: cost ledger,
           \texttt{tree.json}, and \texttt{results.json}. Solid
           edges are control; dashed edges are state.}
  \label{fig:fullstack}
\end{figure*}

\Cref{fig:fullstack} draws the full ARI stack at v0.8.0. The system
splits into one driver and many tool providers. The driver
(\texttt{ari-core}) owns the orchestrator, the BFTS engine, the
lineage walker, the agent loop, the checkpoint, the cost tracker,
and the configuration loader. Each skill is a separate Python
package shipped as a \term{Model Context Protocol} (MCP) server
with its own \texttt{skill.yaml} declaration. MCP is the
JSON-RPC-over-stdio interface that Anthropic standardised for
exposing tool-style capabilities to an LLM agent loop; ARI uses it
as the universal seam between \texttt{ari-core} and every skill. The fourteen skills currently in tree are:
\texttt{benchmark}, \texttt{coding}, \texttt{evaluator},
\texttt{hpc}, \texttt{idea}, \texttt{memory}, \texttt{orchestrator},
\texttt{paper}, \texttt{paper-re}, \texttt{plot}, \texttt{replicate},
\texttt{transform}, \texttt{vlm}, and \texttt{web}.

The orchestrator dispatches calls but does not perform domain work
itself. This separation matters because skills can be added or
swapped without touching the search algorithm: the BFTS engine is
unaware of which skills participate; everything is mediated by the
MCP dispatch layer.

The smaller diagram in \cref{fig:architecture} below is the
``operational'' view (control + state outputs) used in the rest of
the report; \cref{fig:fullstack} is the ``layered'' view (driver →
MCP → skills → persistence).

\subsection{Pipeline DAG}\label{sec:overview:pipeline}

\begin{figure}[t]
  \centering
  \resizebox{\columnwidth}{!}{\input{../shared/figures/tikz/F05_pipeline_dag.tex}}
  \caption{Pipeline DAG: idea $\to$ exploration $\to$ paper $\to$
           review, with \term{refine} closing the back edge. Dashed
           outputs are persisted to the checkpoint; the refine
           cycle rewrites the draft and re-runs the review.}
  \label{fig:pipeline}
\end{figure}

A run reduces to a four-station pipeline (\cref{fig:pipeline}). The
idea station selects or generates research questions; the
exploration station spawns nodes under BFTS; the paper station
compiles the best lineage into a draft; and the review station
grades the draft against the rubric. The graph is a DAG plus one
back edge from review to paper: only this edge can introduce
non-monotonicity (refining worsens an axis), and the
\term{convergence criterion} (\cref{sec:paper:loop}) is what bounds
the back-edge count.

The pipeline is driven by a single runner in the orchestrator; a
scientific-data assembler closes the gap between exploration output
and paper input, collating the per-node artefacts the writer needs.

\subsection{Checkpoint scoping and the design principles}\label{sec:overview:principles}

ARI is built around five \term{design principles} (P1--P5). The
binding one is \textbf{P2: determinism}. Every randomised step
records its seed; every LLM call records its model identity,
prompt, and output; every tool call is tagged with its node identity
(a copy-on-write guard) so its writes land inside the right node's
working tree. Re-running
a checkpoint must reproduce the same artefacts byte-for-byte, except
for fields explicitly marked as wallclock or as model-side
non-determinism.

The other principles, in summary:

\begin{itemize}
  \item \textbf{P1 single artefact tree}: every output of a run lives
        below \texttt{\$ARI\_CHECKPOINT\_DIR}. Nothing leaks to
        \texttt{\$HOME} or \texttt{/tmp}.
  \item \textbf{P3 small components}: each skill is a separate
        package; replacing one skill does not require rebuilding
        \texttt{ari-core}.
  \item \textbf{P4 explicit lineage}: every node and every checkpoint
        record their parent, so a derivative experiment can recompute
        only its delta.
  \item \textbf{P5 cost transparency}: the cost tracker attaches
        dollar costs to every model call so the user can bound a run.
\end{itemize}

The checkpoint serialiser writes three top-level files: the full
search tree to \texttt{tree.json}, per-node payloads to
\texttt{nodes\_tree.json}, and end-of-run aggregates to
\texttt{results.json}. Lineage walks across checkpoints climb from
a child to its ancestors.

  \caption{Checkpoint directory layout. Run-level files (left band)

\Cref{fig:checkpoint} draws the on-disk shape of a checkpoint:
shared run-level files on the left, per-node working trees on the

as input (a follow-up run, a regression check, a paper draft)
points at a single such directory.
