\subsection{Pipeline overview}\label{sec:paper:overview}

  \caption{Paper-generation swim lane: idea, experiment, write. Each
           lane corresponds to a phase; the back edge from

The paper-generation pipeline is the second tree of activity in ARI
(\cref{fig:swim}). Once exploration terminates, the surviving lineage
is handed to the \term{paper} skill, which produces a draft, and the
\term{review} skill, which scores the draft. The implementation is
\texttt{run\_pipeline} in
\code{ari-core/ari/pipeline/orchestrator.py:131}, and the
\texttt{build\_scientific\_data} helper on line~68 of the same file
collates the per-node artefacts the paper skill needs.

A run produces three persisted artefacts: a \texttt{draft.tex} (the
LaTeX paper proper), a \texttt{review.json} (per-leaf rubric scores
and rationale), and a \texttt{rubric.json} (the rubric instance used
to score this run, useful when the rubric is generated on-the-fly).

\subsection{Rubric formalism}\label{sec:paper:rubric}

We model the \term{rubric} $\Rubric$ as a finite tree whose leaves
carry tuples
$(\rubricitem{i},\rubricweight{i},\judgefn{i})$. Each leaf $i$ has
a category descriptor $\rubricitem{i}$, a non-negative weight
$\rubricweight{i}$ with $\sum_i \rubricweight{i}=1$, and a
\emph{judge function} $\judgefn{i}:\paper\to[0,1]$ that returns a
graded score. The paper score is the convex combination
\begin{equation}\label{eq:paperscore}
  \paperscore{\paper} \;=\; \sum_i \rubricweight{i}\,\judgefn{i}(\paper).
\end{equation}
This formalism is shared with the PaperBench rubric tree, which
ARI reuses directly through a vendored package (\cref{sec:paper:paperbench}):
the leaf type \texttt{TaskNode} comes from
\texttt{paperbench.rubric.tasks}, the graded leaf type
\texttt{GradedTaskNode} from
\texttt{paperbench.judge.graded\_task\_node}, and the weighted
aggregation envelope from
\texttt{aggregate\_graded\_tree} in
\code{ari-skill-paper-re/src/\_paperbench\_bridge.py:75}.

Two grader families are used in production: an LLM-backed grader,
implemented for ARI by wrapping PaperBench's \texttt{SimpleJudge}
(\code{ari-skill-paper-re/src/\_paperbench\_bridge.py:54}); and a
deterministic grader for items with a binary or numeric verifier
(e.g.\ ``reproducibility'', ``cost reported in absolute dollars'').
Both implement the \texttt{Evaluator} protocol
(\code{ari-core/ari/protocols/evaluator.py:19}).

\subsubsection*{Venue-conditioned rubric templates}

The rubric generator (\texttt{ari-skill-replicate.generate\_rubric})
accepts an optional venue identifier that selects a YAML template
from \code{ari-core/config/paperbench\_rubrics/}.  The template's
\texttt{prompt\_overrides} block is injected into the skeleton and
subtree prompts via dedicated placeholders; ARI core remains
domain-agnostic (P4), and venue knowledge is confined to YAML.  This
mirrors the existing \code{ari-core/config/reviewer\_rubrics/}
pattern that drives \texttt{ari-skill-paper}'s peer-review path.

Two modes are supported.  In \texttt{agent\_benchmark} mode the
skeleton pass decomposes the paper by scientific structure (one
direct child per major contribution) and leaves grade whether a
candidate submission's \texttt{reproduce.sh} output reproduces the
paper --- the original PaperBench framing.  In \texttt{paper\_audit}
mode the rubric root's direct children are a \emph{fixed} set of
audit axes declared in the YAML's \texttt{top\_level\_axes} list,
and leaves grade descriptive completeness of the paper text (plus
optional Artifact Description / Artifact Evaluation appendices)
rather than submission output --- a framing inversion targeted at
HPC reproducibility-audit research, where the question is no longer
``can an agent reproduce this paper?'' but ``does this paper
describe enough to be reproducible?''.  Templates shipped with the
current release are \texttt{generic} (back-compat), \texttt{sc}
(six axes: environment / data / execution / figures / scaling /
conclusion), \texttt{neurips} (NeurIPS Reproducibility Checklist
axes), and \texttt{nature} (wet-lab Reporting Summary axes).  The
\texttt{paper\_audit} mode requires the two-stage generator path
because a single-pass prompt cannot honour the fixed-axis
constraint.

\subsubsection*{Paper-audit mechanisms (v0.7.2)}

Three orthogonal mechanisms compose to lift the \texttt{paper\_audit}
score from a low single-digit baseline to a regime where the rubric
discriminates papers along the reproducibility axes named above.
All three are pure ARI wrapper layers --- the vendored PaperBench
\texttt{SimpleJudge} is unchanged --- and each is independently
toggleable so the contribution of each mechanism can be ablated.
(An earlier ``Step~4 reproduction-package generator'' that drafted
an LLM-simulated \texttt{reproduce.log} from the paper's own
reported numbers was removed as off-protocol: it short-circuited
PaperBench's executed-submission stage and inflated the audit
score in a way not comparable to leaderboard runs.)

\begin{enumerate}
  \item \emph{Multimodal markdown image expander}
    (\code{ari-skill-paper-re/src/\_litellm\_completer.py}).  The
    paper PDF is converted to markdown by
    \texttt{pymupdf4llm.to\_markdown(write\_images=True)} which
    emits \texttt{!\hspace{0pt}[\hspace{0pt}]\hspace{0pt}(path)}
    references to per-page PNGs.  An expander in
    \texttt{async\_completion} rewrites those references into
    OpenAI multimodal content blocks before the LiteLLM call, so
    vision-capable judges (Claude 4.6/4.7, GPT-5, Gemini 2.5) see
    figures alongside text in the same context.  Vendor
    \texttt{SimpleJudge} is unchanged --- it still embeds
    \texttt{paper\_md} as a text Path; the multimodal translation
    happens at the LiteLLM boundary so the vendor-swap property
    holds.

  \item \emph{Paper-audit prompt patch} in
    \code{ari-skill-paper-re/src/\_paperbench\_bridge.py}.  Vendor
    PaperBench's \texttt{TASK\_CATEGORY\_QUESTIONS} ask
    submission-oriented questions (``does the code contain X?'',
    ``did \texttt{reproduce.sh} produce evidence matching Y?'') for
    each of \texttt{Code Development} / \texttt{Code Execution} /
    \texttt{Result Analysis}.  In paper-audit mode with an empty
    submission those questions structurally collapse to ``no'',
    capping the mean score around $0.3$ even when AD/AE are
    concatenated.  An at-call-scope monkey-patch swaps these for
    paper-audit-flavoured questions (``does the paper describe X
    with concrete specificity?'', ``is the paper's claim Y
    internally consistent with the experimental evidence presented
    elsewhere?'') and restores the originals on exit, so
    \texttt{agent\_benchmark} calls in the same process are
    unaffected.

  \item \emph{Stepwise input conditions}.  The dogfood script
    \code{scripts/sc\_paper\_dogfood.py} supports
    \texttt{-{}-paper-extras AD.pdf AE.pdf} which concatenates
    additional PDFs into the main \texttt{paper.md} so the same
    rubric can be scored under \texttt{paper-only},
    \texttt{paper+AD}, and \texttt{paper+AD+AE} conditions.  This
    is the input axis required to measure the relative
    contribution of each artifact layer.
\end{enumerate}

The three mechanisms compose super-additively: the lift from
combining all three exceeds the sum of their independent lifts,
with the largest single-mechanism contribution coming from the
prompt patch (mechanism 2) and the second from AD/AE concatenation
(mechanism 3); the multimodal expander (mechanism 1) provides a
small but consistent paper-agnostic boost.

\subsubsection*{3-stage bridge contract (v0.7.3)}\label{sec:paper:bridge}

Where paper-audit mode targets the paper itself, the
\term{3-stage bridge contract} (\code{ari-skill-paper-re/src/\_paperbench\_bridge.py})
exposes the protocol-faithful PaperBench loop --- agent rollout,
reproduction, and grading --- as three keyword-only async callables
sharing one calling vocabulary:
\code{rollout\_submission(paper\_md, work\_dir, agent\_model, sandbox\_kind, container\_image, iterative\_agent, env, agent\_env\_path, forbid\_host\_filesystem, blacklist\_urls, time\_limit\_sec, \ldots)},
\code{reproduce\_submission(submission\_dir, sandbox\_kind, container\_image, partition, gpus\_per\_task, gpu\_type, memory\_gb\_per\_node, exclusive, extra\_sbatch\_args, capture\_tarball, salvage\_retries, retry\_threshold\_sec, time\_limit\_sec)},
and \code{judge\_submission(paper\_md, rubric, submission\_dir, reproduce\_log, judge\_model, paper\_audit\_mode, code\_only, \ldots)}.
Internally each wraps the existing single-stage worker
(\code{\_replicator\_agent.run\_replicator\_agent} for Stage~1,
\code{server.run\_reproduce} for Stage~2, vendor \texttt{SimpleJudge}
for Stage~3), so the bridge is a public surface, not a
re-implementation. The deliberate-design choices that make it
vendor-faithful are:

\begin{itemize}
  \item \emph{container\_image alias resolution} --- short aliases
    \texttt{pb-env} / \texttt{pb-reproducer} expand to the canonical
    \texttt{image:latest} tags produced by
    \code{scripts/build\_pb\_images.sh}; URIs / paths / arbitrary
    tags pass through verbatim so user images keep working.
  \item \emph{Fail-loud preconditions} --- when a Stage~2 request
    cannot be served (missing docker daemon / apptainer / sbatch
    / partition / GRES), a \texttt{RuntimeError} is raised rather
    than silently downgrading to host CPU. Two opt-in env vars
    (\texttt{ARI\_PHASE1\_ALLOW\_FALLBACK=1},
    \texttt{ARI\_SLURM\_ALLOW\_NO\_GRES=1}) preserve the legacy
    silent-fallback for sites that need it.
  \item \emph{code\_only consistency} --- Stage~1 instructions
    default to \texttt{code\_only=True} (the agent is asked to write
    code, not execute it). The bridge's \code{judge\_submission}
    accepts \texttt{code\_only} and mirrors vendor
    \code{paperbench/grade.py:109-112}'s
    \code{TaskNode.code\_only()} pruning so the grader's scope
    matches the agent's. Auto-enabled when no \texttt{reproduce.log}
    is present.
  \item \emph{Salvage retries} --- opt-in vendor-style
    \code{reproduce\_on\_computer\_with\_salvaging} (mirror of
    \code{paperbench/reproduce.py:252}) wrapping early-failure
    attempts in a Python~3.11 venv prelude. Total wall-clock budget
    is honoured across attempts (retry-time exclusion).
  \item \emph{Executed-submission tarball} --- every Stage~2 call
    writes \texttt{submission\_executed\_<UTC>.tar.gz} alongside
    the submission so a run is re-gradable later, matching vendor
    \code{paperbench/reproduce.py:230-244}.
  \item \emph{Host-FS source-removal guard} --- caller-toggle
    \texttt{forbid\_host\_filesystem=True} refuses
    \texttt{sandbox\_kind=local} / \texttt{slurm} for
    benchmark-fair runs (both leak the host workspace to the
    agent's bash tool).
  \item \emph{agent.env injection} --- \texttt{agent\_env\_path}
    (auto-discovered from \texttt{ARI\_AGENT\_ENV\_PATH} or
    \texttt{\~{}/.ari/agent.env}) loads per-paper credentials into
    the agent's env; the calling process's \texttt{HF\_TOKEN} is
    automatically forwarded (vendor
    \code{nano/eval.py:172-179} pattern).
  \item \emph{Blacklist URL enforcement} ---
    \texttt{blacklist\_urls} prepends a \texttt{FORBIDDEN URLS}
    block to the agent instruction prompt and exports
    \texttt{ARI\_BLACKLIST\_URLS} for downstream tool wrappers.
\end{itemize}

\subsection{Review/refine loop}\label{sec:paper:loop}

The refine cycle iteratively rewrites $\paper_t$ to $\paper_{t+1}$
under feedback $J(\paper_t)$:
\begin{equation}\label{eq:refine}
  \paper_{t+1} \;=\; \refine\!\bigl(\paper_t, J(\paper_t)\bigr).
\end{equation}

The cycle stops when the \term{convergence criterion}
$\paperscore{\paper_{t+1}}-\paperscore{\paper_t}<\epsilon$ holds for
two successive iterations, or when a per-pipeline cap of three
iterations is reached. The empirical frequency at which each branch
fires --- convergence vs.\ cap --- is what \cref{chapter:eval} is
meant to report once a frozen checkpoint is in place; we make no
quantitative claim here.

The refine step is intentionally non-monotonic per axis: a fix to a
clarity issue may slightly degrade a numeric axis. This is the source
of the back edge in \cref{fig:swim}; the convergence criterion is on
the aggregate score, not per-axis.

\subsection{PaperBench integration via \texttt{paper-re}}\label{sec:paper:paperbench}

The \term{paper-re} skill (\texttt{ari-skill-paper-re}) is ARI's
PaperBench-compatible replication driver. The integration is
\emph{not} a re-implementation: PaperBench is shipped vendored at
\texttt{vendor/paperbench} so that ARI tracks upstream task and
rubric semantics verbatim (the layout requirement is documented in
PaperBench's \texttt{rubric.md \S{}8}). Three integration points
deserve mention.

\subsubsection*{1. Solver subclass: \texttt{AriPBSolver}}

\code{ari-skill-paper-re/src/\_replicator\_agent.py:269} subclasses
PaperBench's upstream
\texttt{paperbench.solvers.basic.BasicAgentSolver}. The chz
immutable-config rule requires the decorator on the subclass even
when no fields are added; ARI's overrides are minimal:

\begin{itemize}
  \item \texttt{\_get\_tools()}: every non-submit tool is wrapped in
        \texttt{\_BoundedOutputTool} so that bash / python /
        file-read / search outputs are size-capped before they flow
        into the next API call. \texttt{submit} is dispatched by
        name and never has its \texttt{execute()} invoked, so it is
        passed through unwrapped.
  \item \texttt{\_run\_agent()}: bypasses
        \texttt{paperbench.solvers.utils.sanity\_check\_docker} via
        \texttt{\_bypass\_docker\_sanity\_check}, and rewrites the
        upstream hardcoded \texttt{/home/\{submission,paper\}}
        paths to ARI's workspace-relative layout (\texttt{LocalComputer}
        cwd = \texttt{repro\_sandbox/}, with paper at
        \texttt{paper/} and submission at \texttt{submission/}).
        Prompt content otherwise comes verbatim from upstream's
        \texttt{get\_instructions} / \texttt{get\_system\_message},
        keeping ARI aligned with the upstream replication-task
        framing (7-day budget, ``do not stop until you have
        replicated all results'', etc.).
\end{itemize}

\texttt{AriPBSolver} also retains upstream's
\texttt{check\_for\_existing\_run} so a partially completed run can
be resumed idempotently.

\subsubsection*{2. Rubric-driven artefact list}

Vendor PaperBench is unaware of ARI's per-paper rubric tree, so
\texttt{AriPBSolver.\_run\_agent} appends an
\texttt{EXPECTED\_ARTIFACTS} block to the user instructions when
\texttt{LocalPBTask.rubric\_expected\_artifacts} is non-empty. The
appended block lists the paths a successful \texttt{reproduce.sh}
must produce; without it, the grader's leaf claims would have no
hint about which files to verify. The rubric tree itself is
constructed by
\code{ari-skill-paper-re/src/\_paperbench\_bridge.py:63}
(\texttt{task\_node\_from\_dict}).

\subsubsection*{3. Driver: \texttt{run\_replicator\_agent}}

\code{ari-skill-paper-re/src/\_replicator\_agent.py:365} is the
async entry point. It returns the standard
\texttt{build\_reproduce\_sh} envelope
$\{$\texttt{populated},
\texttt{output\_dir}, \texttt{files},
\texttt{expected\_artifacts}, \texttt{max\_runtime\_sec},
\texttt{model}, \texttt{prompt\_sha256}, \texttt{notes},
\texttt{warnings}$\}$. The \texttt{output\_dir} is both the agent's
workspace and where ARI expects \texttt{reproduce.sh} after the
rollout; the upstream solver writes \texttt{agent.log} and other
bookkeeping into \texttt{run\_dir}, which we point at the workspace
itself so artefacts live alongside the reproduce script.

The default completer is
\texttt{OpenAIResponsesTurnCompleterConfig} with model
\texttt{gpt-5-mini} (overridable via the
\texttt{ARI\_MODEL\_REPLICATOR} environment variable); LiteLLM-routed
alternatives are wired through the helpers in
\texttt{\_litellm\_completer.py}. The default time limit is twelve
hours per run; \texttt{sandbox\_kind} is \texttt{auto}, falling back
to a local Apptainer image when one is supplied.

\subsubsection*{4. Grading and aggregation}

After the replication agent completes, grading is delegated through
\texttt{judge\_submission} in
\code{ari-skill-paper-re/src/\_paperbench\_bridge.py}. The function
adapts ARI's
$($\texttt{paper\_md}, \texttt{submission\_dir},
\texttt{reproduce\_log}, \texttt{judge\_model}$)$ tuple to upstream
\texttt{SimpleJudge}, which produces a \texttt{GradedTaskNode}
tree per submission. Aggregation across runs is then handled by
two helpers:

\begin{itemize}
  \item \texttt{aggregate\_graded\_tree} (line~75): collapses one
        \texttt{GradedTaskNode} into a weighted scalar via the
        rubric weights $\rubricweight{i}$, plus per-category
        breakdown.
  \item \texttt{average\_graded\_runs} (line~91 region): takes
        the per-run mean over $n$ \texttt{GradedTaskNode} trees, so
        a single replication score is reported even when the agent
        was rerun for noise reduction.
\end{itemize}

The two-stage design --- vendor's \texttt{SimpleJudge} for leaf
grading, ARI's adapters for shape and aggregation --- means that
upgrading PaperBench upstream is a vendor swap (no ARI code
changes), while ARI retains full control of how leaf grades are
combined into a single number. The same envelope feeds into
\cref{eq:paperscore} as $\judgefn{i}\in\{0,1\}$ when the leaf is
binary.

\subsection{Domain breadth: the execution-profile contract}\label{sec:paper:paperbench:v072}

\begin{figure}[t]
  \centering
  \resizebox{\columnwidth}{!}{\input{../shared/figures/tikz/F14_execution_profile.tex}}
  \caption{The \texttt{reproduce\_contract.execution\_profile} field
           threads from a single rubric artefact into two consumers:
           the BasicAgent user message (left band, via
           \texttt{\_format\_hpc\_appendix}) and the Phase 2
           \texttt{sbatch} dispatcher (right band, via
           \texttt{\_run\_reproduce\_slurm}). Three runtime probes
           adapt the dispatched flag set to the local SLURM
           deployment, so the same rubric dispatches faithfully on
           heterogeneous clusters without being modified.}
  \label{fig:execution-profile}
\end{figure}

The PaperBench upstream \texttt{BasicAgent} is designed for a
single Docker container with one GPU. To cover papers whose
methodology is fundamentally parallel --- multi-rank MPI kernels,
multi-node training, cluster-specific module environments --- the
\texttt{paper-re} skill extends the rubric contract with an
optional \texttt{execution\_profile} field on every
\texttt{reproduce\_contract} (\cref{fig:execution-profile}). The
field is domain-agnostic by
design: its \texttt{kind} enum
(\texttt{cpu\_single}, \texttt{gpu\_single}, \texttt{gpu\_multi},
\texttt{mpi}, \texttt{mpi\_gpu}) names a parallel-execution shape
rather than a specific scientific domain, and its sibling fields
declare a paper's scale envelope, expected CSV aggregation
schema, and any SLURM allocation hints. When the rubric generator
judges a paper to be single-machine the field is omitted, and the
pipeline degenerates to the original four-flag \texttt{sbatch}
invocation: the contract is additive, not breaking.

The agent prompt picks up the contract by appending three blocks
to its user message when \texttt{execution\_profile} is
non-empty: the profile itself, a snapshot of the surrounding
SLURM allocation (\texttt{SLURM\_JOB\_NUM\_NODES},
\texttt{SLURM\_NTASKS}, the visible GPU list), and a short
``compute-node execution conventions'' rubric covering shared-FS
path discipline, \texttt{srun}-over-\texttt{mpirun} preference,
Python environment activation, module loads, multi-node fan-out,
and \texttt{timeout} wrapping. A regression test ensures the
inverse never holds: ARI running inside an unrelated SLURM
allocation must not leak SLURM/MPI prose into the prompt of a
non-HPC paper.

The same contract feeds the Phase 2 dispatcher. Beyond the
existing partition / time / cpu-count flags, \texttt{paper-re}
now constructs \texttt{sbatch} commands carrying the full
placement set (nodes, ntasks, ntasks-per-node, nodelist, exclude,
exclusive), GPU set (gpus-per-task, gpus-per-node, GRES type),
memory set (per-node, per-CPU), and hardware-constraint set
(constraint, NUMA bindings, hint), plus an escape hatch for
arbitrary pass-through flags. Three runtime probes guard against
heterogeneous SLURM deployments: a shared-filesystem warning, a
GRES-availability probe that drops every GPU-related flag on
clusters that report no GRES, and an \texttt{sbatch --help} probe
that drops version-incompatible flags (e.g.\ \texttt{--cpu-bind}
on SLURM versions that accept it only on \texttt{srun}).
Together these mean that the same rubric can dispatch faithfully
on a multi-V100 cluster with GRES, on a sandbox partition without
GRES, or on a single-node workstation, with no change to the
rubric itself.

The wrapper layer is the only thing that changes: the vendored
\texttt{paperbench} tree and the \texttt{ari-core} package are
unchanged across this expansion. The visible end-user surface ---
the same paper-registry / wizard / results view as the
v0.7.1 GUI, with an execution-profile override panel added to
the wizard's reproduce step --- is described in
\cref{sec:paper:paperbench}.

\subsection{Failure modes and guardrails}\label{sec:paper:failures}

Three failure modes are common enough to be named:

\begin{itemize}
  \item \textbf{LLM-only support}: the paper makes a claim that no
        node in the lineage substantiates. The mitigation is a static
        check: every numerical or asserted-causal claim must be
        traceable via a \texttt{\textbackslash code\{...\}} reference
        to a node artefact.
  \item \textbf{Citation hallucination}: the paper cites a non-existent
        work. The mitigation is structural; we forbid the LLM from
        inventing citation keys and require all bib entries to be
        verified against Semantic Scholar (\cref{sec:related}).
  \item \textbf{Refine drift}: the score climbs but the paper loses
        coherence. The mitigation is the per-axis floor in the
        rubric: any axis falling below $0.5$ aborts the refine.
\end{itemize}

Source-file selection at draft time uses
\code{node\_selection.py:204}
(\texttt{select\_source\_files\_for\_publication}); this is what
guards against the LLM-only-support failure by enumerating exactly
which lineage nodes a claim can lean on. The companion
\texttt{load\_selected\_sources} (line~258) reads bytes for every
\texttt{(node\_id, rel\_path)} pair and respects an optional
\texttt{size\_budget} so the paper draft never exceeds context.

\subsection{Tool-output bounding for the replicator}\label{sec:paper:bounded-tool}

The replicator agent can spend hours inside a single bash session;
its tool outputs (long stdout, large directory listings) would
otherwise dominate every subsequent prompt. ARI bounds them with
\texttt{\_BoundedOutputTool}
(\code{ari-skill-paper-re/src/\_replicator\_agent.py:243}), which
wraps the inner tool's \texttt{execute()} and post-processes the
output through \texttt{\_truncate\_tool\_output} (line~218) at a
configurable byte limit. This is the difference between PaperBench
upstream's free-form transcript and ARI's bounded, replayable one;
it is also why the prompt budget remains predictable across
multi-hour rollouts.
