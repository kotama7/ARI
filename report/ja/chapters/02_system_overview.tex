\subsection{フルスタック}\label{sec:overview:layout}

\begin{figure*}[t]
  \centering
  \resizebox{0.95\textwidth}{!}{\input{../shared/figures/tikz/F10_full_stack.tex}}
  \caption{v0.7.2 における ARI のフルスタック。CLI / SDK 入口は
           単一のオーケストレータ(\texttt{ari-core})を駆動する;
           オーケストレータは三つのエンジン(BFTS、系譜歩行器、
           ReAct \texttt{AgentLoop})を所有し、MCP 層を通じて 14
           スキルレジストリへ分配する。永続化はすべて単一のチェ

           \texttt{tree.json}、\texttt{results.json}。実線は制御、
           破線は状態を表す。}
  \label{fig:fullstack}
\end{figure*}

\Cref{fig:fullstack} は v0.7.2 の ARI フルスタックを描く。シス
テムは一つのドライバと多数のツール提供者に分かれる。ドライバ
(\texttt{ari-core})はオーケストレータ、BFTS エンジン、系譜歩
行器、エージェントループ、チェックポイント、コストトラッカ、設
定ローダを保有する。各スキルは独立した Python パッケージとして
MCP サーバ形式で出荷され、固有の \texttt{skill.yaml} 宣言を持つ。
現在ツリーに含まれる 14 のスキル:\texttt{benchmark}、
\texttt{coding}、\texttt{evaluator}、\texttt{hpc}、
\texttt{idea}、\texttt{memory}、\texttt{orchestrator}、
\texttt{paper}、\texttt{paper-re}、\texttt{plot}、
\texttt{replicate}、\texttt{transform}、\texttt{vlm}、
\texttt{web}。

オーケストレータは呼び出しの分配のみを担い、ドメイン作業自体は
実行しない。この分離が重要なのは、スキルを追加・差し替えても探
索アルゴリズムには触れずに済むからである:
\code{ari-core/ari/orchestrator/bfts.py:114} の BFTS コードは、
どのスキルが参加するかを知らない;すべては MCP 分配層を介する。

\Cref{fig:architecture} は本レポートの残りで使う「運用ビュー」
(制御 + 状態出力)であり、\cref{fig:fullstack} は「層別ビュー」
(ドライバ → MCP → スキル → 永続化)である。

\subsection{パイプライン DAG}\label{sec:overview:pipeline}

\begin{figure}[t]
  \centering
  \resizebox{\columnwidth}{!}{\input{../shared/figures/tikz/F05_pipeline_dag.tex}}
  \caption{パイプライン DAG: idea $\to$ exploration $\to$ paper
           $\to$ review、ただし \term{改稿} がバックエッジを閉じ
           る。破線出力はチェックポイントへ永続化される;改稿サ
           イクルは草稿を書き直し、レビューを再実行する。}
  \label{fig:pipeline}
\end{figure}

一回の実行は四つのステーションのパイプラインに帰着する
(\cref{fig:pipeline})。idea ステーションは研究問題を選択または
生成し、exploration ステーションは BFTS の下でノードを派生させ、
paper ステーションは最良系譜を草稿に編成し、review ステーション
はルブリックで草稿を採点する。グラフは DAG にレビューから紙への
バックエッジを 1 本加えた構造である:このエッジのみが非単調性を
生じうる(改稿が軸を悪化させ得る);\term{収束基準}
(\cref{sec:paper:loop})がバックエッジ回数の上界を与える。

パイプライン実行器は
\code{ari-core/ari/pipeline/orchestrator.py:131}
(\texttt{run\_pipeline});探索出力と論文入力の間隙を埋める科学
データ組立器は同ファイル 68 行目の
\texttt{build\_scientific\_data} である。

\subsection{チェックポイントスコープと設計原則}\label{sec:overview:principles}

ARI は 5 つの\term{設計原則} (P1--P5) を中心に構築されている。最
も拘束的なのが \textbf{P2:決定性} である。すべての乱択ステップ
は乱数シードを記録し、すべての LLM 呼び出しはモデル同一性、プロ
ンプト、出力を記録し、すべてのツール呼び出しは
\texttt{cow\_node\_id} を伴うことで、書き込みが正しいノードの作
業ツリーへ落ちることを保証する。チェックポイントを再走させれば、
時計依存またはモデル側非決定性として明示的にマークされた領域を除
き、バイト単位で同一の成果物を再現しなければならない。

その他の原則は次の通り:

\begin{itemize}
  \item \textbf{P1 単一成果物ツリー}:実行のすべての出力は
        \texttt{\$ARI\_CHECKPOINT\_DIR} 配下に存在する。
        \texttt{\$HOME} や \texttt{/tmp} へ漏出しない。
  \item \textbf{P3 小コンポーネント}:各スキルは独立パッケージで
        ある;スキル交換のために \texttt{ari-core} を再構築する
        必要はない。
  \item \textbf{P4 明示的系譜}:各ノードと各チェックポイントは
        親を記録するため、派生実験は差分のみを再計算できる。
  \item \textbf{P5 コスト透明性}:\texttt{cost\_tracker.py} は各
        モデル呼び出しにドルコストを付与し、利用者が実行を予算で
        制限できるようにする。
\end{itemize}

チェックポイント直列化器は 3 つのトップレベルファイルを書き出す。
完全な探索木は \texttt{checkpoint.py:49} を経て
\texttt{tree.json} に入り、ノード単位のペイロードは
\texttt{checkpoint.py:59} を経て \texttt{nodes\_tree.json} に入
り、実行末の集計は \texttt{checkpoint.py:64} を経て
\texttt{results.json} に入る。チェックポイントを跨ぐ系譜歩行は
\texttt{lineage.py:126}(\texttt{walk\_ancestor\_ckpts})が担い、
子から先祖へと辿る。
