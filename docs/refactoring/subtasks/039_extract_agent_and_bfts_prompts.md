## 1. Goal

Finish externalizing the **agent ReAct-loop** and **BFTS orchestrator** prompt
text. The *outer* skeletons are already lifted into `ari-core/ari/prompts/`
(`agent/system.md`, `orchestrator/bfts_select.md`,
`orchestrator/bfts_expand_select.md`, `orchestrator/bfts_expand.md`), but a
large amount of **instruction prose still lives inline** as Python string
literals in `ari-core/ari/agent/loop.py` (1630 lines) and
`ari-core/ari/orchestrator/bfts.py` (845 lines):

- the per-node memory-rules block, the root-node and child-node user prompts,
  the node-label task descriptions, and the environment/budget hint prose in
  `loop.py`;
- the `expand()` context-block scaffolds (siblings / ancestors / existing
  children / diversity / depth-budget notes) that `bfts.py` renders in Python
  and injects into `bfts_expand.md` as pre-filled `{..._block}` placeholders.

The goal is to move that **static prose** into externalized `.md` templates
under `ari-core/ari/prompts/agent/` and `ari-core/ari/prompts/orchestrator/`,
**byte-for-byte identical** in rendered output, so that: (a) prompt wording is
editable without touching Python; (b) every agent/BFTS prompt is covered by the
existing sha256 snapshot regression (`ari-core/tests/test_prompt_extraction.py`)
and Gate 10 (`report/scripts/check_prompt_snapshots.py`); and (c) later
provenance work (subtask 044) can hash these prompts. Determinism (design
principle **P2**) is preserved because the rendered bytes do not change.

  1170-line `run` method or the BFTS strategy/IO tangle is subtask **011**

## 6. Current Problems

Grounded, repository-specific:

1. **The outer skeleton is externalized but the body is not.**
   `orchestrator/bfts_expand.md` accepts fifteen placeholders
   (`{goal_line}`, `{sci_note}`, `{depth_note}`, `{budget_note}`,
   `{idea_block}`, `{parent_report_block}`, `{siblings_block}`,
   `{ancestors_block}`, `{existing_block}`, `{diversity_block}`, …), and
   `bfts.py:604-740` constructs every one of those blocks as prose-in-Python
   before injecting it. So editing the *actual instruction the planner reads*
   (e.g. "Already-spawned children of THIS parent (do NOT duplicate these
   directions; propose something complementary)" at `bfts.py:710-712`, or the
   label-saturation rule at `bfts.py:704-707`) still means editing Python, and
   that prose is invisible to the snapshot regression and Gate 10.

2. **Whole prompts remain inline in `loop.py`.** The child-node prompt
   (`loop.py:587-608`, ~22 lines, including the MANDATORY/STERILE contract prose)
   and the root-node prompt (`loop.py:611-621`, ~11 lines, "START NOW … WORKFLOW
   ORDER") are hardcoded string literals. The `_MEMORY_RULES_PER_NODE` block
   (`loop.py:60-68`) is a module-level triple-quoted prompt injected into the
   system prompt as `{memory_rules}` (formatted at `loop.py:551`). None of these
   are hash-pinned; a stray edit shifts LLM behaviour silently.

3. **Node-label semantics are buried in a dict.** `_label_desc`
   (`loop.py:574-580`) maps `improve` / `ablation` / `validation` / `debug` /
   `draft` to one-line task descriptions that become part of the child prompt.
   These are behavioural instructions, not code.

4. **Provenance blind spot.** Because these strings are inline, subtask 044
   (prompt-version tracking) cannot hash them: `load_versioned()` only covers
   files under `ari/prompts/`. Until they are externalized, the majority of the
   agent/BFTS prompt surface can never appear in a run's provenance record.

5. **Reviewer friction.** `loop.py` (1630 L) and `bfts.py` (845 L) are large;
   a prompt-wording tweak is indistinguishable from a logic change in the diff,
   and lives hundreds of lines from the load site.

## 7. Proposed Design / Policy

### 7.1 Classification (using the master vocabulary)

Apply the prompt-specialized vocabulary from
`011_prompt_management_plan.md` §5.x — **EXTRACT_TEMPLATE /
MOVE_TO_CONFIGURABLE_PROMPT / KEEP_INLINE / REVIEW_REQUIRED** — to each target.
(MERGE / MOVE_TO_LEGACY / DELETE_CANDIDATE do not apply: nothing here duplicates
a core template or is dead.)

**EXTRACT_TEMPLATE** — the whole static prose moves to a new `.md`, filled by
the same `str.format` values already computed in Python:

| Source (inline) | Proposed key | Placeholders (from current code) |
| --- | --- | --- |
| `loop.py:60-68` `_MEMORY_RULES_PER_NODE` | `agent/memory_rules` | `{node_id}` |
| `loop.py:611-621` root `user_content` | `agent/user_root` | `{goal_text}`, `{node_id}`, `{depth}`, `{first_tool}` |
| `loop.py:587-608` child `user_content` | `agent/user_child` | `{goal_text}`, `{node_id}`, `{depth}`, `{label}`, `{label_desc}`, `{workflow_hint}` |
| `bfts.py:704-719` existing-children duplication + saturation prose | `orchestrator/bfts_expand_existing_children` | `{existing_lines}`, `{label_dist_str}`, `{threshold}`, `{saturated}` |

**MOVE_TO_CONFIGURABLE_PROMPT** — extract the *static scaffold* to a small
fragment template; the surrounding Python keeps computing the dynamic,
conditional values and passes them in (this is exactly how `bfts_expand.md`
already consumes `{siblings_block}` etc.):

- `loop.py:494-548` environment/budget hints → `agent/env_budget_hint`
  fragment(s). The Python that decides whether a SLURM/container/provided-files
  clause is present stays; only the fixed prose ("EXPERIMENT ENVIRONMENT:",
  "RESOURCE BUDGET:", "Do NOT build, pull, or switch container images — reuse
  this one.") moves to the fragment.
- `bfts.py:604-665, 721-740` block scaffolds (`sci_note`, `depth_note`,
  `budget_note`, `idea_block`, `siblings_block`, `ancestors_block`,
  `diversity_block`) → small `orchestrator/bfts_expand_*` fragments. Each
  fragment holds the fixed heading/prose (e.g. "Sibling scores at same depth:",
  "Tree diversity so far:") with `{...}` placeholders for the per-line data the
  Python loop assembles. The per-node fact loops (e.g. `sibling_lines` at
  `bfts.py:637-645`) remain in Python.
- `loop.py:277-281` `PLATFORM NOTE` block → `agent/platform_note` fragment
  (`{partition}`, `{missing}` placeholders); the probe-reading Python stays.

**KEEP_INLINE** (do not extract — pure data serialization, not instruction
prose):

- `bfts.py:451-470` and `bfts.py:537-550` `candidate_descriptions` — these emit
  per-node fact lines (`[{i}] id=…, depth=…, label=…, metrics=…`). The fixed
  substrings are field labels for a data record, not LLM instructions; the
  cost/benefit of templating a formatting loop is poor. **REVIEW_REQUIRED** if
  036's census argues otherwise.
- `loop.py:213, 323, 348` single-line bracket labels in
  `build_working_context_messages` ("[Experiment context (stable across all
  nodes):]", "[Established conclusions from ancestor nodes (N):]", "[Related
  prior findings from ancestors (N):]") — one-line context headers, not
  standalone prompts. Recommend KEEP; **REVIEW_REQUIRED** if bundled.

**REVIEW_REQUIRED (design):**

- `_label_desc` (`loop.py:574-580`): five short label→description strings.
  Option A — one keyed template `agent/label_descriptions.md` rendered by
  label; Option B — leave as a Python data map (it is a lookup table, not a
  prompt). Recommend a small externalized data file *only if* 037
  (`define_prompt_template_policy`) blesses a "keyed fragment" pattern;
  otherwise KEEP. Decide with 036/037 in hand.

### 7.2 Composition mechanism (no new machinery)

- New fragments are loaded with the **existing** `FilesystemPromptLoader`
  (lazy in-function import, matching the current convention at `loop.py:51`,
  `bfts.py:475/553/743`).
- A fragment that is composed into a parent template (e.g. an
  `orchestrator/bfts_expand_*` block folded into
  `{siblings_block}` of `bfts_expand.md`) is loaded, `.format(...)`-filled with
  the per-line data, and passed as the parent placeholder value — **the parent
  `bfts_expand.md` is unchanged** (its placeholder names and bytes stay, so its
  pinned hash at `test_prompt_extraction.py:53` stays green).
- Whole-prompt templates (`agent/user_root`, `agent/user_child`,
  `agent/memory_rules`) replace the inline literal: the code loads the key and
  `.format(...)`s it with the same variables it interpolated before.

### 7.3 Byte-identical guarantee (the invariant)

For every extraction, the rendered string sent to the LLM (`system_content` at
`loop.py:554`; `user_content` at `loop.py:587`/`:611`; the expand prompt at
`bfts.py:744-760`) must be **byte-for-byte identical** before and after. The
mechanism (mirrors the `agent/system` extraction discipline in PC3):

1. Capture the sha256 of the *rendered* literal (with a canned variable set)
   from the current `main` before moving it.
2. After moving, render the new template with the same variables and assert the
   sha256 is unchanged.
3. Pin the raw-template sha256 in `_EXPECTED_HASHES` and pin the rendered-fixture
   sha256 in a new assertion (Section 12).

Watch items that historically bit this repo: trailing-newline handling (see the
`extract_metrics` note at `test_prompt_extraction.py:55-60`) and the
`str.format` brace-escaping already present in `agent/system.md` (`{{ … }}` for
literal JSON braces). Any literal `{`/`}` in extracted prose must be doubled in
the `.md` exactly as the Python literal did, or the render will differ.

### 7.4 Directory / key convention

Follow the existing two-tier convention (`011_prompt_management_plan.md` §6.1):
core prompts live at `ari-core/ari/prompts/<category>/<name>.md`, `<category>`
mirroring the consuming module. New agent prompts → `agent/`; new BFTS prompts →
`orchestrator/`. No skill directories are touched (039 is core-only).

## 11. Compatibility Constraints

1. **Byte-identical rendered output** is the hard invariant (Section 7.3). If a
   diff appears, the extraction is wrong — fix the template, do not update the
   pinned hash to match a drifted value.
2. **Same commit, three artifacts.** Each extraction PR must land, together:
   the template `.md`, the `_EXPECTED_HASHES` row, and the regenerated report
   snapshot. Otherwise `test_prompt_extraction.py` or Gate 10 fails CI
   (`.github/workflows/refactor-guards.yml`, `docs-change-coupling.yml`).

   `ari/prompts/**/README.md` enumerations; the readme-sync gate
   (`scripts/readme_sync.py`, `.github/workflows/readme-sync.yml`) may require
   these to stay in sync — update them in the same PR.
4. **Brace escaping.** Literal `{`/`}` inside extracted prose must be doubled in
   the `.md` (as `agent/system.md` already does for JSON examples), or
   `str.format` will raise / mis-render.

   existing hashes and add a runtime dependency (`011_prompt_management_plan.md`
   §6.2). Deferred.
6. **Coordinate with subtask 011.** `separate_bfts_strategy_from_react_loop`
   (Phase 3) restructures the same `bfts.py`/`loop.py` line ranges. The

## 14. Rollback Plan

Low-cost and self-contained (prompt relocation only):

1. **Per-template revert.** Each template is added in an isolated commit paired
   with its `_EXPECTED_HASHES` row and snapshot; `git revert` of that commit
   restores the inline literal and drops the row/snapshot atomically.
2. **Full revert.** Because 039 changes only `loop.py`, `bfts.py`, new `.md`
   files, the snapshot test, the README enumerations, and regenerated report
   snapshots — with **no** schema/API/behaviour change — reverting the whole
   PR returns the tree to a byte-identical runtime state.
3. **Safety net is the invariant.** If a regression surfaces post-merge, the
   rendered-fixture hash pinpoints the drifted template; revert that one commit.
   No data migration, no checkpoint format change, nothing stateful to undo.
4. **Snapshot desync recovery.** If Gate 10 fails after a partial revert, re-run
   `scripts/snapshot_prompts.py` (`make snapshot-prompts`) to resync
   `report/shared/appendix/prompts/` — never hand-edit either side
   (`report/CLAUDE.md` §2).

## 17. Notes for Implementer

- **Naming caveat (read first):** the master plan
  `011_prompt_management_plan.md` §13 labels *its* "039" as evaluator judge
  prompts. Ignore that local numbering — the **index** (`007_subtask_index.md`)
  is authoritative and assigns 039 = agent + BFTS. Evaluator = 040, paper =
  041. Do not extract evaluator/paper prompts here.
- **Mechanical discipline, not creative rewriting.** Copy the literal into the
  `.md` unchanged, then verify the render hash. If you feel tempted to "clean
  up" wording, stop — that is a separate, human-reviewed change, not 039.
- **Double every literal brace.** `agent/system.md` shows the pattern
  (`{{"status": "success", …}}` for a JSON example). The child prompt and any
  extracted block with `{`/`}` must double them exactly as the Python literal
  did.
- **Watch trailing newlines.** The `extract_metrics` note
  (`test_prompt_extraction.py:55-60`) documents a real prior mismatch where the
  `.md` gained a trailing `\n` the constant lacked and the call site stripped
  one. Decide per template whether the file ends with `\n` and whether the call
  site strips — and pin the *rendered* hash so the choice is enforced.
- **Fragment composition keeps the parent stable.** For the BFTS blocks, do not
  edit `bfts_expand.md`. Load each `bfts_expand_*` fragment, fill it, and pass
  the result as the existing `{..._block}` value — the parent's pinned hash
  (`test_prompt_extraction.py:53`) must not move.
- **Grep before removing a constant.** Before deleting `_MEMORY_RULES_PER_NODE`
  or `_label_desc`, `grep -rn` them across `ari-core/` (esp. `tests/`); if
  imported, keep a source-compatible shim like `SYSTEM_PROMPT`.
- **Regenerate, never hand-edit, snapshots.** Use `scripts/snapshot_prompts.py`
  / `make snapshot-prompts` for `report/shared/appendix/prompts/`
  (`report/CLAUDE.md` §2). A hand-edited snapshot will pass Gate 10 today and
  rot silently.
- **This is core-only.** No `ari-skill-*` package is touched; the ari-skill →
  ari-core dependency direction is not involved. No frontend, no MCP, no
  dashboard API.
- **Prefer many small commits.** One template family per commit (agent whole
  prompts; agent hints; BFTS existing-children; BFTS blocks) keeps each rollback
  atomic and each rendered-fixture diff readable.
