## 1. 目的と非目的

### 目的

1. **追跡可能性** — 各 doc が「どのソースファイルを文書化しているか」を明示し、リファクタや
   仕様変更時にどの doc を見直すべきかを機械的に逆引きできるようにする。
2. **stale 検出の土台** — 分割・削除されたモジュール(例: `ari-core/ari/cli.py`,
   `ari-core/ari/pipeline.py` は v0.7.1 で分割済)への参照が docs に残る事故を、CI で検出する。
3. **既存ゲートの形式化** — `release_policy.md` §4 /「`refactor_audit.md` §11-4」が要求する
   「文書化された CLI フラグ・env・REST・MCP ツールは実ソースに対応すること」を、宣言+検証の
   仕組みに落とす。

### 非目的(本設計では扱わない)

- セクション単位の細粒度注記(Layer 2)— 将来拡張(§7)。
- stale/drift 検出(git 更新時刻比較)— 将来拡張(§7)。今回の検証は **path 実在のみ**。
- doc 本文の加筆・正確性レビュー — 機構導入とは独立。
- 実装(検証スクリプト作成、front-matter の一括付与)— 別途承認後(§6 段階導入)。

## 3. front-matter スキーマ

各 doc の**先頭**に YAML front-matter を置く。GitHub の `.md` レンダリングは有効な
front-matter を非表示にするため、本文の見た目には影響しない。

```yaml
---
sources:
  - path: ari-core/ari/orchestrator/bfts.py     # リポジトリ根からの相対パス(必須)
    role: implementation                         # 任意。下記語彙
  - path: ari-core/config/workflow.yaml
    role: config
generated: false                                 # 任意。自動生成doc なら true(既定 false)
last_verified: 2026-05-25                         # 任意。最終確認日(ISO 8601)
---
```

### フィールド定義

| キー | 必須 | 型 | 意味 |
|---|:--:|---|---|
| `sources` | ✓(ライブ doc) | list | この doc が文書化するソースの一覧。空不可 |
| `sources[].path` | ✓ | str | **リポジトリ根**からの相対パス。ファイル or ディレクトリ |
| `sources[].role` | – | enum | `implementation` / `schema` / `config` / `prompt` / `test` / `vendor` / `doc` |
| `generated` | – | bool | doc が自動生成物なら `true`(既定 `false`) |
| `last_verified` | – | date | 人手で対応を確認した日付 |

### `role` 語彙

| role | 対象例 |
|---|---|
| `implementation` | `*.py` 実装(`bfts.py`, `server.py` など) |
| `schema` | `*.schema.json`(`replication_rubric.schema.json` など) |
| `config` | `workflow.yaml`, `defaults.yaml`, `model_prices.yaml` |
| `prompt` | `ari/prompts/**`, skill 側 `src/prompts/**` |
| `test` | `tests/**`(契約をテストがピンしている場合) |
| `vendor` | `vendor/**`(PaperBench 等) |
| `doc` | 他 doc / `README.md` / `CHANGELOG.md`(doc が doc に由来する場合) |

### 配置ルール

- **ライブ doc**(`getting-started/` `concepts/` `guides/` `reference/` `about/`)は `sources` 必須。
- **免除リスト**(`sources` 不要): `docs/README.md`、`docs/リファクタリング計画書.md`、本設計書、
  `docs/_archive/**`(歴史的成果物)。検証スクリプトの除外設定に列挙する。
- **翻訳(`ja/` `zh/`)**: 原則 en の同名 doc と**同一の `sources`** を持つ。将来的に
  「翻訳は対応する en doc を `role: doc` で指す」方式も可(§7)。当面は en をミラー。

## 4. 検証スクリプト `scripts/docs/check_doc_sources.py`(仕様)

実装は本設計の範囲外。仕様のみ定義する。

### 入力

- 走査対象: `docs/**/*.md`(既定。`--lang en|all` で範囲指定)。
- リポジトリ根: スクリプト位置から自動解決。

### チェック内容(すべて満たさなければ失敗)

1. **YAML パース可能**: front-matter があれば正しい YAML である。
2. **必須宣言**: ライブ doc(免除リスト外)は `sources` を持ち、空でない。
3. **path 実在**: 各 `sources[].path` がリポジトリ根からの相対で**実在**する
   (ファイル or ディレクトリ)。← 本機構の中核。
4. **role 妥当**: `role` がある場合、§3 の語彙のいずれか。
5. **(任意警告)** `generated: true` の doc は `sources` に generator スクリプトを含むことを推奨。

### 出力・終了コード

- 失敗ごとに `「<doc>: <path>」 が見つからない` 等を1行で列挙。
- 1件でも失敗で **exit 1**(CI を落とす)。全通過で exit 0。
- `--json` で機械可読出力(他ツール連携用)。

### CI 組み込み

`release_policy.md` の docs ゲート(Release checklist §4)に1項追加:

> - `python scripts/docs/check_doc_sources.py` が exit 0(全 doc の宣言ソースが実在)。

`refactor_audit.md` §11-4 のゲートと整合する(「文書化対象は実シンボルに対応」)。

## 7. 将来拡張(本設計の範囲外)

- **Layer 2 — セクション単位注記**: 主要節に `> 📄 **Source:** \`path\`` の可視バッジ。
  細粒度の追跡が必要になったら導入。既存の散在した `Source:` 行はこの形式に正規化。
- **stale/drift 検出**: `git log -1 --format=%ct` でソース更新時刻と doc 更新時刻を比較し、
  「ソースが新しい=要更新」を警告。S1 のスクリプトに `--check-drift` として追加可能。
- **逆引きインデックス生成**: front-matter を集約し「ソース → それを説明する doc」の
  逆引き表を自動生成(`docs/README.md` か `docs/_meta/source_index.md`)。
- **サイト表示**: `docs.html` の Markdown ローダが front-matter を**剥がして**表示する
  3行の前処理を追加(生の `---` ヘッダがブラウザに出ないように)。GitHub リポジトリ表示は
  自動で非表示になるため、こちらが主要消費経路なら優先度は低い。

## 8. 意思決定(確定 / 2026-05-25)

| # | 論点 | 決定 |
|---:|---|---|
| 1 | 記録方式 | **front-matter**(各 doc 先頭の YAML)。中央マニフェスト・可視フッターは不採用 |
| 2 | 粒度・検証 | **ファイル単位 + path 実在検査**。セクション単位/ drift は将来拡張 |
| 3 | 導入範囲 | **設計提案のみ**(本書まで)。実装は §6 を承認後に着手 |

### 次アクション

本設計は確定済み・実装未着手。実装に進む場合は §6 の **S1(検証スクリプト)+ S2(見本1件)**
から着手するのが安全。指示があれば着手する。
