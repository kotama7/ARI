## 0. 目的と非目的

### 目的

1. **発見性の向上** — フラットなルート直下に概念/手順/リファレンスが混在している現状を、
   Diátaxis(Tutorial / How-to / Reference / Explanation)に沿って分類する。
2. **入口の明確化** — `docs/` に Markdown の目次(`README.md`)を新設し、HTML サイト
   (`index.html` / `docs.html`)に依存しない閲覧導線を確立する。
3. **多言語パリティの可視化と是正** — en/ja/zh の欠落を一覧化し、翻訳ステータスを
   manifest で管理する。
4. **命名規約の統一** — 大文字小文字の揺れ(`PHILOSOPHY.md` のみ大文字)を解消する。

### 非目的(本計画では扱わない)

- 各ドキュメントの**内容の書き換え/加筆**(構造移動とリンク追従に限定。誤記修正は別PR)。
- `ari-core` 等ソースコードのリファクタリング(別管理:`docs/_archive/refactor_audit.md` 参照)。
- HTML サイトのデザイン刷新(ナビ href とアンカーの追従のみ行う)。

## 4. フェーズ計画

各フェーズは独立PRとし、フェーズ末尾の受け入れゲート(§5)を通過してから次へ進む。

### Phase 1 — 骨格と目次の新設(非破壊・追加のみ)

- 目標ツリーの空ディレクトリ + `docs/README.md`(目次・パリティ表・命名規約・言語切替)を新設。
- 既存ファイルは**まだ動かさない**。リンク切れゼロを保証。
- 成果物: `docs/README.md`, 空ディレクトリ(`.gitkeep`)。

### Phase 2 — en の移動とリンク追従(本体)

1. §3 の対応表に従い **`git mv`** で29ファイルを移動。
2. 156本の `.md` リンクを機械置換(下記スクリプト)。相対形式・`docs/`プレフィックス両対応。
3. 文中のアンカー(`#...`)はファイル内容を変えないため不変。ただしリンク先パスのみ更新。
4. 旧パスに**リダイレクト・スタブ**(1行ポインタ)を一時設置するか否かは §7 で決定
   (外部ブックマーク/README 参照のため、被参照上位=`quickstart`/`registry`/`philosophy` のみ
   スタブ設置→次マイナーで削除、を推奨)。

リンク書き換えの基本方針(`git grep` で全件抽出 → sed マッピング → 目視 diff):

```bash
# 1. 旧→新 の対応を1ファイルにし、各 .md / .html に適用
# 2. 置換後に必ず「壊れたリンク検査」(§5 gate)を実行
# 3. ari-core 側からの docs 参照(docs/xxx.md)も同時に追従(リポジトリ全体 grep)
git grep -nE '\]\([^)]*\.md' -- 'docs/**'           # docs 内リンク
git grep -nE 'docs/[A-Za-z_/]+\.md'                  # ソース/README からの参照
```

> ⚠ リポジトリ根の `README.md` / `README.ja.md` / `README.zh.md` / `CHANGELOG.md` /
> `CONTRIBUTING.md` も docs パスを参照している可能性が高い。Phase 2 のリンク追従対象に含める。

### Phase 3 — HTML サイトと i18n の追従

- `docs.html` サイドバーの10 href、`index.html` の参照を新パスへ更新(§1-3)。
- アンカー参照(`reference/mcp_tools.md#...` 等)もパス部のみ更新、フラグメントは不変。
- i18n の `id` スパンはテキストのみで、リンクは HTML 側 href のため、`i18n/*.js` の変更は原則不要
  (ナビ項目の追加が無ければ)。

### Phase 3b — メディア集約(任意)

- `images/` `movie/` `logo.png` `sample_paper.pdf` を `assets/` 配下へ集約。
- HTML(`src="logo.png"`, `sample_paper.pdf`)と各 `.md` の画像参照を追従。
- リスク高・効果中のため**任意/後続**に位置づける(本体構造化には必須ではない)。

### Phase 4 — ja/zh のミラー移動とパリティ是正

1. 既存19ファイルを en と同一構造へ `git mv`(Phase 2 と同じ対応表)。
2. 各言語10欠落ファイル(§1-4)を**今回まとめて翻訳して埋める**(§7-4 で確定。スタブ暫定にしない)。
   対象(ja/zh 各10): root の `release_policy` / `refactor_audit`、guides の
   `migration` / `testing` / `troubleshooting`、reference の `environment_variables` /
   `file_formats` / `mcp_tools` / `public_api` / `rest_api`。計 **20 ファイルの新規翻訳**。
   - 翻訳は en 確定版(Phase 2 後)を原本とし、構造・見出し・リンクを en とミラーする。
   - コード/コマンド/環境変数名/識別子は翻訳しない(技術用語は原綴り維持)。
3. 翻訳ステータス manifest(`docs/README.md` 内のパリティ表)を真実の唯一の情報源とし、
   是正後は全セルが ✓ になることをゲート4で確認する。

### Phase 5 — 巨大ドキュメントの分割(任意・後続)

- `concepts/architecture.md`(約1150行)は概念+リファレンスが混在。将来的に

  156リンク網への影響が大きいため**本計画の必須範囲外**。別計画として切り出す。

### Phase 6 — クリーンアップ

- 受け入れゲート全通過の確認、`docs/README.md` の最終化。

## 5. 受け入れゲート(各フェーズ末尾で実行)

`docs/about/release_policy.md` の docs ゲートを踏襲・拡張する。

1. **壊れたリンク=0**: 全 `.md` の相対リンク・`docs/`プレフィックスリンクが実在ファイルに解決する
   (抽出 → 解決チェックのスクリプトを CI 化)。
2. **HTML href 解決=0破壊**: `index.html` / `docs.html` の `.md`/メディア href が実在する。
3. **`~/.ari/` 残渣=0**(既存ゲート維持): `grep -rn '~/\.ari/' docs/`(`_archive/` 除く)が空。
4. **パリティ表整合**: `docs/README.md` のパリティ表が実ファイルと一致(✗ は manifest に記録)。
5. **孤立ファイル=0**: `docs/README.md` の目次から全ライブ `.md` へ到達可能。
6. **`git mv` 検証**: `git log --follow` で履歴が連続(コピー&削除ではなく移動)。

## 6. リスクと緩和

| リスク | 影響 | 緩和 |
|---|---|---|
| HTMLサイトのナビ崩壊(§1-3) | 公開ドキュメントが404 | Phase 3 で href 全件追従 + ゲート2 |
| 156リンクの追従漏れ | リンク切れ | スクリプト一括置換 + ゲート1 を CI 化 |
| 外部ブックマーク/README 切れ | 流入断 | 被参照上位のみ一時スタブ(§7決定) |
| ja/zh の追従漏れ | 言語別リンク切れ | Phase 4 を en と同一対応表で機械適用 |
| ソース側 `docs/...` 参照切れ | ドキュメント誘導切れ | Phase 2 でリポジトリ全体 grep を対象に含める |
| kebab-case 化による書き換え増 | 工数増 | §7 で snake_case 維持案と比較し決定 |

**ロールバック**: 全変更は `bfts` から切る作業ブランチ上で `git mv`/sed により実施。
問題時は `git restore` / ブランチ破棄で即時復旧。コミットはフェーズ単位で分割し、
個別リバートを可能にする。
